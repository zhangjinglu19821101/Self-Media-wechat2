
# 智能化改进 - 简明实施路径

## 一句话总结

**去掉能力边界判定层，执行 Agent 直接执行，通过标准化格式和上游约束，让 Agent B 完全基于 LLM 智能决策。**

---

## 一、三个核心改变（立即可以做）

### 改变 1：去掉能力边界判定层（最大的改变！）

**当前做法**：
```typescript
// 先判定能力边界
const capabilityCheckResult = await this.callExecutorAgent(task, previousResult);

// 再决定是否真正执行
if (!capabilityCheckResult.isNeedMcp &amp;&amp; capabilityCheckResult.isTaskDown) {
  const directResult = await this.callExecutorAgentDirectly(task, previousResult);
}
```

**新做法**：
```typescript
// 直接执行，不做能力判定！
const executorResult = await this.callExecutorAgentDirectly(task, previousResult);
```

**为什么这样做**：
- 备份文件1告诉我们：LLM 可以基于接口文档自主判断
- 备份文件2告诉我们：标准化流程比复杂判断更重要
- 当前问题：能力判定经常出错，反而耽误事

---

### 改变 2：执行 Agent 返回标准化格式（上游约束！）

**当前问题**：
- 执行 Agent 返回格式不统一
- 有时候保存 null，有时候保存对象
- Agent B 解析时经常出错

**新要求**：执行 Agent 必须返回固定格式

```typescript
interface ExecutorStandardResult {
  // 核心状态（4选1）
  status: 'COMPLETED' | 'PARTIAL' | 'BLOCKED' | 'FAILED';
  
  // 执行结果（如果 COMPLETED）
  result?: any;
  
  // 问题描述（如果不是 COMPLETED）
  problem?: {
    type: 'CAPABILITY_LIMIT' | 'MISSING_INFO' | 'TECHNICAL_ERROR' | 'NEED_CONFIRM';
    description: string;
    context?: Record&lt;string, any&gt;;
  };
  
  // 元数据（必须有）
  metadata: {
    executionTime: number;
    timestamp: string;
    agentVersion: string;
  };
}
```

**验证器（在源头就检查！）**：
```typescript
function validateExecutorResult(result: any): { isValid: boolean; errors: string[] } {
  const errors: string[] = [];
  
  // 只检查格式，不检查内容！
  if (!result.status) errors.push('缺少 status 字段');
  if (!result.metadata) errors.push('缺少 metadata 字段');
  // ... 更多格式检查
  
  return { isValid: errors.length === 0, errors };
}
```

### 改变 3：Agent B 完全基于 LLM 决策（最大化智能！）

**当前做法**：
- 三层硬编码检查（execution_result 存在性、内容长度、关键词匹配）
- 固定的决策流程
- 很多 if-else 判断

**新做法**：
```typescript
async agentBIntelligentDecision(task, executorResult) {
  // 1. 构建完整上下文
  const context = buildExecutionContext(task, executorResult);
  
  // 2. 构建提示词（给 LLM 足够信息）
  const prompt = buildAgentBPrompt(context);
  
  // 3. 调用 LLM（单次调用，完全信任！）
  const llmResponse = await callLLM('Agent B', '智能决策', prompt);
  
  // 4. 解析并验证格式
  const decision = JSON.parse(llmResponse);
  const validation = validateAgentBDecision(decision);
  
  // 5. 执行决策
  if (validation.isValid) {
    await executeDecision(task, decision);
  } else {
    // 降级方案：转人工
    await markAsNeedUser(task, validation.errors);
  }
}
```

**提示词设计原则**（从备份文件1学来的）：
- 给 LLM 完整的信息（任务、执行结果、历史、可用能力）
- 明确角色定位和职责
- 给出输出格式要求
- **不做具体规则指导，让 LLM 自主决策！**

## 四、风险和缓解措施

| 风险 | 影响 | 缓解措施 |
|-----|------|---------|
| 去掉能力判定层后，执行 Agent 可能频繁失败 | 中 | 保留完整的历史记录，Agent B 可以基于历史决策 |
| LLM 决策不稳定 | 高 | 完善降级方案，格式验证失败时立即转人工 |
| 执行 Agent 不按标准格式返回 | 高 | 在源头就验证格式，不合格要求重执行 |
| 业务逻辑改变影响现有流程 | 中 | 分阶段迁移，充分测试，保留回滚方案 |
