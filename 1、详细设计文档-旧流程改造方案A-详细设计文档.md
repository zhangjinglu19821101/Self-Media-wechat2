### 3.2 `callAgentBWithDecision()` 方法（核心决策）

**文件位置**: `src/lib/services/subtask-execution-engine.ts`  
**行号**: 2099-2250

#### 3.2.1 真实调用链（关键！）

```typescript
public async callAgentBWithDecision(
  task: typeof agentSubTasks.$inferSelect,
  executionContext: ExecutionContext,
  capabilities: any[]
): Promise&lt;AgentBDecision&gt; {
  
  // ========== 关键：直接调用 callLLM() ==========
  
  // 1. 构建提示词（包含合规检查指导）
  const prompt = `
    你是 Agent B，技术专家，拥有不断完善的MCP能力，负责推动各个执行agent解决他们遇到的所有技术问题。
    
    【重要说明】
    如果任务涉及保险事业部的合规检查（insurance-d），必须严格执行以下要求：
    1. 只要任务是合规检查，必须使用 wechat_compliance/content_audit 或 wechat_compliance/content_audit_simple
    2. 即使执行Agent说已经做过检查，你也要重新执行合规检查，确保合规
    
    【当前执行情况】
    - 是否已做过合规检查：${...}
    - 合规检查是否通过：${...}
    
    ...（其他提示词内容）
  `;
  
  // 2. 单次 LLM 调用
  const response = await callLLM(
    'agent B',
    '标准化决策',
    '你是 Agent B，负责综合多方信息做出标准化决策',
    prompt
  );
  
  // 3. 解析响应为 AgentBDecision
  const decision: AgentBDecision = JSON.parse(response);
  
  return decision;
}
```

## 四、决策流程处理

### 4.1 决策类型处理（实际实现）

```typescript
switch (agentBDecision.type) {
  case 'COMPLETE':
    await this.handleCompleteDecision(...);
    return;
    
  case 'NEED_USER':
    await this.handleNeedUserDecision(...);
    return;
    
  case 'FAILED':
    await this.handleFailedDecision(...);
    return;
    
  case 'EXECUTE_MCP':
    const mcpSuccess = await this.executeMcpWithRetry(...);
    if (mcpSuccess) {
      continue;  // 继续下一轮决策
    } else {
      continue;  // 继续下一轮决策
    }
}
```

### 4.2 决策类型对比（架构图 vs 实际）

| 架构图决策类型 | 实际决策类型 | 映射逻辑 |
|-------------|------------|---------|
| APPROVE | COMPLETE | ❌ 无映射（直接使用 COMPLETE） |
| NEED_REVISE | NEED_USER | ❌ 无映射（直接使用 NEED_USER） |
| NEED_USER | NEED_USER | ✅ 直接使用 |
| EXECUTE_MCP | EXECUTE_MCP | ✅ 直接使用 |
| CAN_HELP | NEED_USER | ❌ 无映射（不使用 CAN_HELP） |

## 五、合规检查的实际实现方式

### 5.1 实现原理

**完全基于提示词 + Agent B 自主决策，无硬编码逻辑！**

```
┌─────────────────────────────────────────────────────────────┐
│                    合规检查实现方式                          │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  1. 提示词中包含合规检查指导                                │
│     └─ "如果涉及保险事业部合规检查，必须严格执行"           │
│                                                              │
│  2. 提示词中包含当前执行情况检查                            │
│     └─ 是否已做过合规检查？合规是否通过？                    │
│                                                              │
│  3. Agent B 根据提示词自主决定下一步                        │
│     └─ 第1轮：决策 EXECUTE_MCP（合规检查）                  │
│     └─ 第2轮：决策 COMPLETE（任务完成）                     │
│                                                              │
│  4. 完全依赖 Agent B 的 LLM 智能                            │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 优势与劣势

| 优势 | 劣势 |
|-----|------|
| 代码简洁，无硬编码逻辑 | 完全依赖 LLM 质量，不可控 |
| 灵活性高，Agent B 可自主调整 | 调试困难，无法预知决策路径 |
| 提示词可动态调整 | 缺少代码级别的二次检查 |
