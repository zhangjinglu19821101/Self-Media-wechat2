# 状态机改造设计方案

## 📋 概述

基于现有代码，参考备份代码的优秀设计，进行状态机改造。核心目标是：**明确执行Agent和Agent B的职责边界，Agent B作为技术专家，仅在执行Agent无法完成时才介入**。

---

## 🎯 核心设计原则

### 1. 职责清晰划分
- **执行Agent**：首先尝试独立完成任务，只有在确实无法完成时才求助Agent B
- **Agent B**：技术专家，仅在执行Agent无法完成时介入，提供技术支持

### 2. 状态机设计
- 保持备份代码的清晰状态设计
- 保留 `pre_completed` 和 `pre_need_support` 作为中间状态
- 通过中间状态明确区分两个Agent的处理阶段

---

## 📊 状态定义

### 完整状态列表

| 状态 | 说明 | 责任方 |
|--------|------|--------|
| `pending` | 待执行 | - |
| `in_progress` | 执行Agent正在处理 | 执行Agent |
| `pre_completed` | 执行Agent完成，等待Agent B评审 | 等待Agent B |
| `pre_need_support` | 执行Agent需要帮助，等待Agent B评审 | 等待Agent B |
| `completed` | 已完成 | - |
| `waiting_user` | 等待用户 | - |
| `need_support` | 需要支持（兜底） | - |

### 状态流转图

```
阶段1：执行Agent独立处理
  ↓
pending → in_progress
  ↓
  ├─ 能独立完成 → pre_completed → 等待Agent B评审
  └─ 需要帮助 → pre_need_support → 等待Agent B评审

阶段2：Agent B评审
  ↓
pre_completed → Agent B评审
  ↓
  ├─ APPROVE → completed
  ├─ NEED_REVISE → pending（回到执行Agent重新执行）
  └─ NEED_USER → waiting_user

pre_need_support → Agent B评审
  ↓
  ├─ CAN_HELP → pending（回到执行Agent，带着Agent B的帮助）
  └─ NEED_USER → waiting_user
```

## 🏗️ 核心架构设计

### 1. 执行Agent处理模块

#### 方法：`handlePendingStatus`
**职责**：处理 `pending` 状态，执行Agent独立处理任务

**流程**：
```
步骤1：更新状态为 in_progress
步骤2：获取前序结果（链式传递）
步骤3：调用执行Agent直接处理
步骤4：根据结果更新状态
  ├─ 能完成 → pre_completed
  └─ 需要帮助 → pre_need_support
```

**输入参数**：
- `task`: 当前任务
- `allTasksInGroup`: 同一组的所有任务（用于链式传递）

**输出**：
- 更新任务状态为 `pre_completed` 或 `pre_need_support`

---

### 2. Agent B评审模块

#### 方法：`handlePreCompletedStatus`
**职责**：处理 `pre_completed` 状态，Agent B评审执行Agent的完成结果

**流程（6步严谨设计）**：
```
第一步：准备阶段（验证）
  - 验证任务存在
  - 验证状态正确
  - 验证执行者信息

第二步：记录 Agent B request
  - 获取历史记录
  - 计算下一个交互编号
  - 解析执行Agent的结果
  - 记录 Agent B request

第三步：调用 Agent B
  - 调用 Agent B 进行评审
  - 捕获网络错误、超时错误等

第四步：解析响应
  - 提取 JSON 格式的响应
  - 验证必要字段
  - 验证决策类型

第五步：处理评审结果并记录 response
  - 记录 Agent B response
  - 根据决策类型处理
    ├─ APPROVE → 标记为 completed
    ├─ NEED_REVISE → 回到 pending
    └─ NEED_USER → 标记为 waiting_user

第六步：最后更新状态
  - 所有逻辑都成功了，才更新数据库
  - 确保数据一致性
```

**Agent B决策类型（pre_completed）**：
- `APPROVE`：批准，直接标记为完成
- `NEED_REVISE`：需要修改，让执行Agent重新执行
- `NEED_USER`：需要用户确认

---

#### 方法：`handlePreNeedSupportStatus`
**职责**：处理 `pre_need_support` 状态，Agent B评审执行Agent的求助请求

**流程（同样6步严谨设计）**：
```
第一步：准备阶段（验证）
第二步：记录 Agent B request
第三步：调用 Agent B
第四步：解析响应
第五步：处理评审结果并记录 response
  - 根据决策类型处理
    ├─ CAN_HELP → 回到 pending（带着Agent B的帮助）
    └─ NEED_USER → 标记为 waiting_user
第六步：最后更新状态
```

**Agent B决策类型（pre_need_support）**：
- `CAN_HELP`：可以提供帮助，回到执行Agent
- `NEED_USER`：需要用户确认

## 📝 详细改造方案

### 改造1：状态机主流程改造

**文件**：`src/lib/services/subtask-execution-engine.ts`

**改造内容**：
1. 在 `processGroup` 方法中，根据状态调用不同的处理方法
2. `pending` → 调用 `handlePendingStatus`
3. `pre_completed` → 调用 `handlePreCompletedStatus`
4. `pre_need_support` → 调用 `handlePreNeedSupportStatus`

**伪代码**：
```typescript
private async processGroup(tasks: typeof agentSubTasks.$inferSelect[]) {
  // ... 现有代码 ...

  for (const task of currentStepTasks) {
    switch (task.status) {
      case 'pending':
        await this.handlePendingStatus(task, currentStepTasks);
        return;
      
      case 'pre_completed':
        await this.handlePreCompletedStatus(task);
        return;
      
      case 'pre_need_support':
        await this.handlePreNeedSupportStatus(task);
        return;
    }
  }
}
```

## 🎯 关键设计亮点

### 1. 清晰的职责边界
- 执行Agent：`pending` → `in_progress` → `pre_completed`/`pre_need_support`
- Agent B：`pre_completed`/`pre_need_support` → 评审 → 最终状态

### 2. 严谨的6步流程
- 确保数据一致性
- 所有逻辑都成功了，才更新状态
- 异常处理：不同异常不同处理，状态不回退

### 3. 完整的交互记录
- 记录 Agent B request
- 记录 Agent B response
- 支持历史追溯和审计

### 4. 灵活的决策机制
- `pre_completed`：APPROVE、NEED_REVISE、NEED_USER
- `pre_need_support`：CAN_HELP、NEED_USER
