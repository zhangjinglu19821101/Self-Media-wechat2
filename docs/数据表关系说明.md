## 📋 文档信息

## 🗂️ 一、三张核心任务表

### 1.1 表关系图

```
┌─────────────────────────────────────────────────────────────────────────┐
│                          三张核心任务表关系                                │
└─────────────────────────────────────────────────────────────────────────┘

agentTasks (Agent 任务表 - 总任务)
    │
    │ 1 : N
    │
    ▼
dailyTask (日工作任务表 - 拆分后的任务)
    │
    │ 1 : N
    │
    ▼
agentSubTasks (Agent 子任务表 - 执行步骤)
```

---

## 📊 二、每张表的用途

### 2.1 `agentTasks` - Agent 任务表（总任务）

**用途**: 存储 Agent 间下达的总任务

**关键字段**:
- `taskId`: 任务ID（格式：`master-A-to-B-20260210-001`）
- `coreCommand`: 核心指令内容
- `fromAgentId`: 发起 Agent ID
- `toAgentId`: 接收 Agent ID

**示例**:
- Agent A 给 Agent B 下达一个总任务："写一篇关于保险的文章"

---

### 2.2 `dailyTask` - 日工作任务表（拆分后的任务）

**用途**: 存储拆分后的每日工作任务

**关键字段**:
- `id`: UUID 主键
- `taskId`: 任务ID（格式：`daily-task-{executor}-{date}-{seq}`）
- `relatedTaskId`: 关联总任务ID（外键关联 `agentTasks.taskId`）
- `taskTitle`: 任务标题
- `taskDescription`: 任务描述 ⭐
- `executor`: 执行主体
- `executionDate`: 执行日期

**示例**:
- 总任务"写一篇关于保险的文章"被拆分为：
  1. 收集保险资料
  2. 撰写文章大纲
  3. 完成文章写作
  4. 合规检查
  5. 发布文章

---

### 2.3 `agentSubTasks` - Agent 子任务表（执行步骤）

**用途**: 存储 Agent 自己拆分的执行步骤

**关键字段**:
- `id`: UUID 主键
- `commandResultId`: 关联 `dailyTask.id` ⭐
- `fromParentsExecutor`: 从父 daily_task 继承的 executor
- `taskTitle`: 子任务标题
- `orderIndex`: 执行顺序
- `status`: 状态（`pending` | `in_progress` | `pre_completed` | `pre_need_support` | `waiting_user` | `completed` | `failed`）

**示例**:
- dailyTask"收集保险资料"可能被执行 Agent 拆分为：
  1. 查询保险产品数据库
  2. 整理资料
  3. 保存结果

---

## 🔗 三、表关联关系

### 3.1 关联字段说明

| 关联 | 字段 | 说明 |
|------|------|------|
| `agentTasks` → `dailyTask` | `dailyTask.relatedTaskId` = `agentTasks.taskId` | 一个总任务可以拆分为多个日工作任务 |
| `dailyTask` → `agentSubTasks` | `agentSubTasks.commandResultId` = `dailyTask.id` | 一个日工作任务可以拆分为多个执行步骤 |

## ❓ 四、为什么查询 `dailyTask` 表而不是 `agentSubTasks` 表？

### 4.1 在用户反馈 API 中的代码

**文件**: `src/app/api/agents/user-decision/route.ts`

```typescript
// 3. 查询原始指令（从 daily_task 表）
const task = await db.query.dailyTask.findFirst({
  where: eq(dailyTask.id, commandResultId),  // ⭐ 注意这里！
});
```

### 4.2 原因分析

**变量命名的线索**:

| 变量名 | 关联表 | 说明 |
|--------|--------|------|
| `subTaskId` | `agentSubTasks` | 子任务 ID（agentSubTasks 表的主键） |
| `commandResultId` | `dailyTask` | 指令结果 ID（dailyTask 表的主键）⭐ |

**为什么需要查询 `dailyTask` 表**:

1. **获取原始指令信息**
   - `dailyTask.taskTitle`: 任务标题
   - `dailyTask.taskDescription`: 任务描述（原始指令）
   - 这些信息在 `agentSubTasks` 表中可能不完整或不存在

2. **层级关系**
   - `agentSubTasks` 是 `dailyTask` 的子表
   - `dailyTask` 包含更高级别的任务信息
   - 用户反馈需要知道**整个任务**的上下文，而不仅仅是某个子步骤

3. **数据完整性**
   - `dailyTask` 表包含任务的完整描述
   - `agentSubTasks` 表只包含执行步骤的信息

## 📝 五、快速参考

### 5.1 API 参数与表的对应关系

```typescript
// 用户反馈 API 的参数
{
  subTaskId,        // → agentSubTasks.id (子任务ID)
  commandResultId,  // → dailyTask.id (日工作任务ID) ⭐
  userDecision,     // 用户决策
  decisionType,     // 决策类型
  interactionData,  // 交互数据
}
```

### 5.2 查询顺序

```
1. 先通过 subTaskId 查询 agentSubTasks（验证子任务存在）
   ↓
2. 再通过 commandResultId 查询 dailyTask（获取原始指令信息）⭐
   ↓
3. 记录用户反馈到 agent_sub_tasks_step_history
```
