## 一、评审结论与改进建议

### 1.1 评审结论

| 评审维度 | 评分 | 说明 |
|---------|------|------|
| 业务完整性 | ★★★★☆ | 6种创作类型覆盖主要场景，但缺少「混合创作」模式 |
| 数据结构合理性 | ★★★★☆ | 复用 material_library 是正确选择，但 structuredData 字段需要校验机制 |
| 数据流连贯性 | ★★★☆☆ | 存在3处数据流断裂风险（详见下文） |
| 前端交互设计 | ★★★★☆ | 主+辅素材模式直观，但步骤数偏多，需考虑精简 |
| 与现有系统兼容性 | ★★★★★ | 正交设计，不破坏现有流程 |
| 可扩展性 | ★★★★★ | 新增创作类型/素材类型只需添加配置，无需改代码 |

### 1.2 评审发现的问题与改进

#### 🔴 P0：数据流断裂风险

**问题1：`articleType` 和 `articleLength` 在数据流中未定义传递路径**

现有数据流链路为：`前端 → simple-split API → agent_sub_tasks.metadata → SubtaskExecutionEngine → PromptAssemblerService`

当前 `simple-split/route.ts` 的请求体解构中**没有** `articleType` 和 `articleLength` 字段，metadata 中也**没有**存储这两个字段。如果不补充，创作类型信息会在 API 层丢失。

**改进方案**：

```
前端提交:
  articleType + articleLength + primaryMaterialId + auxiliaryMaterialIds
    ↓
simple-split API:
  解构新增这4个字段
  存入 agent_sub_tasks.metadata:
    { articleType, articleLength, primaryMaterialId, auxiliaryMaterialIds }
    ↓
SubtaskExecutionEngine:
  从 metadata 读取 articleType / articleLength
  查询 primaryMaterial 和 auxiliaryMaterials 的 structuredData
  传入 PromptAssemblyOptions
    ↓
PromptAssemblerService:
  根据 articleType + articleLength 生成结构模板
  格式化素材数据注入提示词
```

**问题2：`primaryMaterialId` 的素材查询需要携带 workspaceId 隔离**

现有 `material_library` 表有 `workspace_id` 隔离，但系统素材（ownerType='system'）的 workspaceId 为 null。执行引擎查询主素材时必须同时查「当前 workspace」和「系统素材」。

**改进方案**：

```typescript
// 查询主素材时
const primaryMaterial = await db
  .select()
  .from(materialLibrary)
  .where(
    and(
      eq(materialLibrary.id, primaryMaterialId),
      or(
        eq(materialLibrary.workspaceId, task.workspaceId),
        eq(materialLibrary.ownerType, 'system')
      )
    )
  );
```

**问题3：`structuredData` 字段无 Schema 校验**

JSONB 字段可以存任何内容，没有类型约束。如果 `type='analogy'` 但 `structuredData` 里存的是法规结构，会导致 AI 提示词注入错误内容。

**改进方案**：

```typescript
// 新增校验服务 src/lib/services/material-structured-data-validator.ts
function validateStructuredData(type: MaterialType, data: unknown): { valid: boolean; errors: string[] }

// 在素材创建/更新 API 中调用校验
// POST /api/materials → validateStructuredData(type, structuredData)
// PUT /api/materials/[id] → validateStructuredData(type, structuredData)
```

#### 🟠 P1：交互设计改进

**问题4：创作类型选择增加用户操作步骤**

当前创作引导已有4个Tab（核心观点/情感基调/素材选择/账号选择），再增加「创作类型」Tab 会让步骤过多。

**改进方案**：将「创作类型」整合到「情感基调」Tab 中，形成「基调与类型」合并 Tab。

```
原方案：核心观点 → 情感基调 → 素材选择 → 账号选择（4步）
改进后：核心观点 → 基调与类型 → 素材选择 → 账号选择（3步）

「基调与类型」合并 Tab:
┌──────────────────────────────────────────────┐
│  创作类型（决定叙事结构）                       │
│  [事件驱动] [客户误区] [新认知] [法规条款]...  │
│                                                │
│  情感基调（决定语气风格）                       │
│  [理性客观] [踩坑警醒] [共情式破局] [专业权威] │
│                                                │
│  💡 选择「客户误区型」时推荐搭配「共情式破局」  │
└──────────────────────────────────────────────┘
```

**问题5：篇幅选择与平台适配存在冗余**

小红书天然是短篇，公众号天然是中长篇。篇幅选择应该由平台自动推荐默认值，用户可覆盖。

**改进方案**：

```typescript
// 平台默认篇幅映射
const PLATFORM_DEFAULT_LENGTH: Record<string, ArticleLength> = {
  wechat_official: 'medium',  // 公众号默认中篇
  xiaohongshu: 'short',       // 小红书默认短篇
  zhihu: 'medium',            // 知乎默认中篇
  douyin: 'short',            // 抖音默认短篇
  weibo: 'short',             // 微博默认短篇
};
```

#### 🟠 P1：业务设计改进

**问题6：「产品测评型」和「投保指南型」的素材结构偏重，前期可简化**

产品测评和投保指南的素材数据结构复杂（多产品对比、方案配置等），前期维护成本高。而且这两类内容的数据变化频繁（产品停售、费率调整），预置素材容易过时。

**改进方案**：Phase 1 只实现4种核心类型（事件驱动/客户误区/行业新认知/法规条款），产品测评和投保指南作为 Phase 2 扩展。Phase 2 时再定义复杂结构。

**问题7：类比素材（analogy）的定位需要明确——它应该是独立的素材库，还是误区素材的子结构？**

当前设计中，analogy 既是素材类型（`type='analogy'`），又包含在误区素材的 `structuredData.analogy` 字段中。这导致同一条类比内容可能存储两次。

**改进方案**：采用「素材引用」模式——误区素材的 `structuredData.analogy` 只存类比素材的 ID，不存内容。运行时通过 ID 查询获取完整内容。

```typescript
// 误区素材的 structuredData.analogy 改为引用
interface MisconceptionStructuredData {
  misconception: { ... };
  breakthrough: { ... };
  
  // 改为引用模式：存类比素材ID，不存内容
  analogyIds: string[];  // 引用类比素材库中的素材
  
  evidence: { ... };
  relatedLawIds: string[];
  relatedCaseIds: string[];
}

// 类比素材单独存储（type='analogy'）
interface AnalogyStructuredData {
  metaphor: string;        // 核心比喻标题
  content: string;         // 完整类比内容
  applicableScenes: string[]; // 适用场景标签
}
```

优势：
- 类比素材独立管理，可被多个误区引用
- 修改类比内容只需改一处
- 类比也可单独用于事件驱动型等场景

#### 🟡 P2：标签体系改进

**问题8：`emotionTags` 同时承载「创作类型」和「情感风格」两种语义，会造成筛选混乱**

当前设计中 `emotionTags: ['共情式破局', '客户误区型']`，前者是情感风格，后者是创作类型，混在一起会导致按标签筛选时逻辑混乱。

**改进方案**：创作类型用 `articleType` 字段存储，不再复用 `emotionTags`。

```typescript
// 改进后的标签分工
topicTags: string[];    // 保险专业维度（重疾险/医疗险/理赔...）
sceneTags: string[];    // 内容用途维度（反驳误区-回本/投保告知/理赔纠纷...）
emotionTags: string[];  // 情感风格维度（共情式破局/理性客观/踩坑警醒...）
// articleType: string  // 创作类型维度（单独字段，不混入标签）
```

## 二、体系总览

### 2.1 核心理念

```
一篇保险科普文章 = 创作类型（骨架） + 素材库（血肉） + 风格模板（外衣） + 平台适配（格式）
```

四个维度**正交组合**，互相独立、灵活搭配：

- **创作类型**：决定文章的叙事结构和节奏（6种）
- **素材库**：提供具体的事件、数据、类比、法规等内容（5库）
- **风格模板**：控制用词、语气、句式等写作风格（已有系统，无需改动）
- **平台适配**：控制输出格式和篇幅（已有系统，无需改动）

### 2.3 关键设计原则

| 原则 | 说明 |
|------|------|
| **创作类型与素材库解耦** | 类型决定结构，素材填充内容，两者独立选择 |
| **素材可跨类型复用** | 同一条法规素材既可用于法规条款型，也可用于事件驱动型作为权威背书 |
| **主素材 + 辅素材模式** | 选1个主素材（决定类型）+ N个辅素材（增强内容），不强制单选 |
| **篇幅与结构联动** | 短篇幅自动合并模块，长篇幅自动展开，与平台适配 |
| **复用现有素材库** | 在 material_library 表上扩展 type 和 structuredData 字段，不新建表 |
| **引用而非复制** | 误区素材引用类比素材ID，运行时查询，避免数据冗余 |
| **默认值智能推荐** | 篇幅由平台自动推荐，创作类型与情感基调联动推荐 |

### 4.4 structuredData 校验规则

```typescript
// 每种素材类型对应的 structuredData 必填字段校验
const STRUCTURED_DATA_REQUIRED_FIELDS: Record<string, string[]> = {
  'case': ['summary', 'coreIssue', 'publicReactions'],
  'misconception': ['misconception', 'breakthrough', 'analogyIds'],
  'analogy': ['metaphor', 'content', 'applicableScenes'],
  'law': ['lawName', 'articleNumber', 'clauseContent'],
  'story': [], // story 类型不做强制校验（通用性最强）
  'data': [],  // data 类型不做强制校验
};

// 校验函数
function validateStructuredData(
  type: MaterialType,
  data: Record<string, unknown>
): { valid: boolean; errors: string[] } {
  const requiredFields = STRUCTURED_DATA_REQUIRED_FIELDS[type] || [];
  const errors: string[] = [];

  for (const field of requiredFields) {
    if (!data[field]) {
      errors.push(`素材类型 "${type}" 的 structuredData 缺少必填字段: ${field}`);
    }
  }

  // 特殊校验：误区素材的 misconception.type 必须是有效值
  if (type === 'misconception' && data.misconception) {
    const validTypes = ['回本逻辑', '别人遭遇', '没用上', '有钱不需要', '隐瞒没事', '跟风买就行', '自定义'];
    const mcType = (data.misconception as Record<string, unknown>).type as string;
    if (!validTypes.includes(mcType)) {
      errors.push(`误区类型 "${mcType}" 不是有效值，有效值: ${validTypes.join('/')}`);
    }
  }

  // 特殊校验：法规素材的 clauseContent 必须有 original 和 plainLanguage
  if (type === 'law' && data.clauseContent) {
    const cc = data.clauseContent as Record<string, unknown>;
    if (!cc.original) errors.push('法规素材的 clauseContent.original（条款原文）为必填');
    if (!cc.plainLanguage) errors.push('法规素材的 clauseContent.plainLanguage（通俗解读）为必填');
  }

  return { valid: errors.length === 0, errors };
}
```

```typescript
/**
 * 根据创作类型和篇幅生成结构模板
 */
formatArticleStructureTemplate(type: ArticleType, length: ArticleLength): string {
  const templates = ARTICLE_STRUCTURE_TEMPLATES[type];
  if (!templates) return '';

  // 根据篇幅合并/省略模块
  const rules = LENGTH_STRUCTURE_RULES[length];
  let template = templates.medium; // 基础模板

  if (length === 'short' && rules.mergeModules.length > 0) {
    // 短篇：合并模块
    for (const [mod1, mod2] of rules.mergeModules) {
      template = template.replace(`【${mod1}】\n...\n\n【${mod2}】`, `【${mod1}+${mod2}】`);
    }
    // 短篇：省略模块
    if (rules.omitModules) {
      for (const mod of rules.omitModules) {
        template = template.replace(new RegExp(`【${mod}】[^【]*`), '');
      }
    }
  }

  return template;
}

/**
 * 格式化素材数据为提示词文本
 * 根据素材类型生成不同格式的注入文本
 */
formatMaterialDataForPrompt(material: MaterialItem): string {
  const sd = material.structuredData;
  if (!sd) return material.content || '';

  switch (material.type) {
    case 'misconception': {
      const mc = sd.misconception as Record<string, string>;
      const bt = sd.breakthrough as Record<string, string>;
      return `
【错误认知】${mc.wrongExample}
【破局逻辑】${bt.standardMismatch}。${bt.essenceClarification}
【价值重构】${bt.essenceClarification}
${sd.evidence?.personalStory ? `【实证案例】${(sd.evidence as Record<string, string>).personalStory}` : ''}
${sd.evidence?.rhetoricalQuestion ? `【灵魂拷问】${(sd.evidence as Record<string, string>).rhetoricalQuestion}` : ''}
      `.trim();
    }

    case 'analogy': {
      const asd = sd as AnalogyStructuredData;
      return `
【类比：${asd.metaphor}】${asd.content}
${asd.valueEquation ? `【价值等式】${asd.valueEquation}` : ''}
      `.trim();
    }

    case 'law': {
      const lsd = sd as LawStructuredData;
      const cc = lsd.clauseContent;
      return `
【法规引用】${lsd.fullReference}
【条款原文】${cc.original}
【通俗解读】${cc.plainLanguage}
【核心要点】${cc.keyPoints.join('；')}
${lsd.companyComparisons?.length ? `【公司条款对比】${lsd.companyComparisons.map(c => `${c.companies.join(' vs ')}：${c.conclusion}`).join('；')}` : ''}
      `.trim();
    }

    case 'case': {
      const esd = sd as EventStructuredData;
      return `
【事件概括】${esd.summary}
【事件经过】${esd.fullStory}
【大众反应】${esd.publicReactions.map(r => `"${r}"`).join('；')}
【核心问题】${esd.coreIssue}
${esd.takeaways?.length ? `【延伸思考】${esd.takeaways.map((t, i) => `${i + 1}. ${t}`).join('\n')}` : ''}
      `.trim();
    }

    default:
      return material.content || '';
  }
}
```

### 7.4 SubtaskExecutionEngine 改造

```typescript
// buildExecutionContext() 新增逻辑

async buildExecutionContext(task: AgentSubTask): Promise<ExecutionContext> {
  const metadata = task.metadata as Record<string, unknown>;
  const ctx: ExecutionContext = {
    // ... 现有字段
  };

  // === 新增：创作类型体系 ===
  const articleType = metadata.articleType as ArticleType | undefined;
  const articleLength = metadata.articleLength as ArticleLength | undefined;
  const primaryMaterialId = metadata.primaryMaterialId as string | undefined;
  const auxiliaryMaterialIds = metadata.auxiliaryMaterialIds as string[] | undefined;

  if (articleType && primaryMaterialId) {
    // 1. 查询主素材（含 workspaceId 隔离）
    const primaryMaterial = await db
      .select()
      .from(materialLibrary)
      .where(
        and(
          eq(materialLibrary.id, primaryMaterialId),
          or(
            eq(materialLibrary.workspaceId, task.workspaceId),
            eq(materialLibrary.ownerType, 'system')
          )
        )
      )
      .limit(1);

    // 2. 查询辅素材
    const auxMaterials = auxiliaryMaterialIds?.length
      ? await db
          .select()
          .from(materialLibrary)
          .where(
            and(
              inArray(materialLibrary.id, auxiliaryMaterialIds),
              or(
                eq(materialLibrary.workspaceId, task.workspaceId),
                eq(materialLibrary.ownerType, 'system')
              )
            )
          )
      : [];

    // 3. 查询误区素材引用的类比素材
    let analogyMaterials: MaterialItem[] = [];
    if (primaryMaterial[0]?.type === 'misconception') {
      const sd = primaryMaterial[0].structuredData as MisconceptionStructuredData;
      if (sd.analogyIds?.length) {
        analogyMaterials = await db
          .select()
          .from(materialLibrary)
          .where(
            and(
              inArray(materialLibrary.id, sd.analogyIds),
              or(
                eq(materialLibrary.workspaceId, task.workspaceId),
                eq(materialLibrary.ownerType, 'system')
              )
            )
          );
      }
    }

    // 4. 注入到 ExecutionContext
    ctx.articleType = articleType;
    ctx.articleLength = articleLength || PLATFORM_DEFAULT_LENGTH[metadata.platform as string] || 'medium';
    ctx.primaryMaterialData = primaryMaterial[0]
      ? this.promptAssembler.formatMaterialDataForPrompt(primaryMaterial[0])
      : '';
    ctx.auxiliaryMaterialData = [
      ...auxMaterials.map(m => this.promptAssembler.formatMaterialDataForPrompt(m)),
      ...analogyMaterials.map(m => this.promptAssembler.formatMaterialDataForPrompt(m)),
    ].join('\n\n');
    ctx.articleStructureTemplate = this.promptAssembler.formatArticleStructureTemplate(
      articleType,
      ctx.articleLength
    );
  }

  return ctx;
}
```

## 十一、风险与对策

| 风险 | 级别 | 对策 |
|------|------|------|
| structuredData 字段过大影响查询性能 | 🟡 | JSONB GIN 索引 + 列表查询时不返回 structuredData（仅详情查询返回） |
| 创作类型选择增加用户认知负担 | 🟡 | 默认不选（走现有流程），选择后增强；联动推荐降低决策成本 |
| 6种类型模板维护成本高 | 🟠 | 模板配置化管理，存储在数据库或配置文件中，不硬编码 |
| 预置素材不够用 | 🟡 | 支持用户自建素材 + Phase 3 的 AI 辅助生成 |
| 类比素材与误区素材的引用关系维护 | 🟡 | 前端提供「引用管理」界面，删除类比时检查引用计数 |
| 产品测评素材数据频繁过时 | 🟠 | Phase 2 再实现，优先保证核心4类型稳定 |
| 同一素材被多种创作类型使用 | 💚 | articleType 字段支持逗号分隔或数组存储，查询时按包含匹配 |
