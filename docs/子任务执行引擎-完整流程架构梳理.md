## 🎯 一、系统架构总览

### 1.1 核心角色定义

| 角色 | 职责 | 关键能力 |
|------|------|----------|
| **控制器 (Controller)** | 任务调度、状态管理、流程协调 | 定时扫描、分组执行、超时处理 |
| **执行Agent (Executor Agent)** | 直接执行任务、自主判断能力边界 | 能力边界判定、直接执行任务 |
| **Agent B (技术专家)** | 解决方案选型、MCP调用决策、用户交互 | 智能决策、MCP参数生成、用户交互 |
| **用户 (User)** | 最终决策者、提供关键信息 | 确认/选择/输入关键信息 |

### 1.2 核心数据表

| 表名 | 用途 | 关键字段 |
|------|------|----------|
| `agent_sub_tasks` | 子任务主表 | `status`, `orderIndex`, `executionResult` |
| `capability_list` | MCP能力清单 | `toolName`, `actionName`, `paramDesc` |
| `agent_sub_tasks_step_history` | 交互步骤历史 | `interactType`, `interactNum`, `interactContent` |
| `agent_sub_tasks_mcp_executions` | MCP执行历史 | `attemptId`, `resultStatus`, `errorCode` |
| `dailyTask` | 主任务表 | `completedSubTasks`, `progress` |

## 🔄 二、完整状态机设计

### 2.1 状态流转图

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                         子任务完整状态机                                        │
└─────────────────────────────────────────────────────────────────────────────────┘

pending (待执行)
    │
    │ [控制器扫描]
    ↓
in_progress (执行中)
    │
    ├─────────────────────────────────────────┐
    │                                         │
    │ [执行Agent自主判定]                     │
    │                                         │
    ↓                                         ↓
pre_completed (执行Agent说搞定了)      pre_need_support (执行Agent需要帮助)
    │                                         │
    │ [Agent B评审]                           │ [Agent B评审]
    ├─→ APPROVED ──→ completed ✅           ├─→ CAN_HELP ──→ pending (重新执行)
    ├─→ NEED_REVISE ─→ pending (重新执行)    ├─→ NEED_USER ──→ waiting_user
    └─→ NEED_USER ──→ waiting_user           └─→ CANNOT_HELP ─→ waiting_user (用户支持)
              │                                           │
              │ [用户反馈后]                              │ [用户反馈后]
              │                                           │
              │  ╔═══════════════════════════════════════════════════════════╗
              │  ║  用户反馈触发流程（API: /api/agents/user-decision）        ║
              │  ╠═══════════════════════════════════════════════════════════╣
              │  ║  1. 用户在前端提交反馈                                      ║
              │  ║  2. API 记录用户交互到 step_history                        ║
              │  ║  3. API 将状态 waiting_user → in_progress                   ║
              │  ║  4. API 调用 manuallyExecuteInProgressSubtasks()           ║
              │  ║  5. 执行引擎 execute() 被触发                               ║
              │  ╚═══════════════════════════════════════════════════════════╝
              │                                           │
              └───────────────┬───────────────────────────┘
                              ↓
                      executeAgentBDecisionAndMcp()
                              │
              ┌───────────────┼───────────────────────┐
              ↓               ↓                       ↓
          COMPLETE        NEED_USER            RE_EXECUTE
              │               │                   (新增！)
              ↓               ↓                       ↓
          completed      waiting_user          pending (重新执行)
              │
              ↓
          FAILED
              │
              ↓
          failed
```

### 2.2 状态详细说明

| 状态 | 触发场景 | 后续动作 |
|------|----------|----------|
| `pending` | 初始状态、重新执行 | 控制器扫描，前置任务完成后执行 |
| `in_progress` | 执行Agent正在处理 | 等待执行Agent返回结果 |
| `pre_completed` | 执行Agent说搞定了 | Agent B三层验证 + 评审 |
| `pre_need_support` | 执行Agent需要帮助 | Agent B评审，决定是否能帮忙 |
| `waiting_user` | 需要用户介入 | 等待用户反馈（确认/选择/输入） |
| `completed` | 任务完成 | 更新主任务进度，触发后续任务 |
| `failed` | 任务失败 | 记录失败，通知相关方 |

### 2.3 用户反馈触发机制（关键补充！）

#### 触发入口：`POST /api/agents/user-decision`

```
┌─────────────────────────────────────────────────────────────────────────┐
│              waiting_user 状态用户反馈完整流程                            │
└─────────────────────────────────────────────────────────────────────────┘

waiting_user (等待用户)
    │
    │ [用户在前端提交反馈]
    │
    ▼
1️⃣ API: POST /api/agents/user-decision
    │
    ├─ 参数:
    │   ├─ subTaskId: 子任务ID
    │   ├─ commandResultId: 指令结果ID
    │   ├─ userDecision: 用户决策/建议
    │   ├─ decisionType: 'waiting_user' | 'retry_failed' | 'redecision'
    │   └─ interactionData: 交互数据（waiting_user场景专用）
    │
    ├─ 验证:
    │   └─ 仅 waiting_user 和 failed 状态可以提交用户决策
    │
    ▼
2️⃣ 记录用户交互
    │
    ├─ 查询沟通历史记录（agent_sub_tasks_step_history）
    ├─ 计算下一个交互编号（nextInteractNum）
    ├─ 根据决策类型构建 interactContent
    │   ├─ waiting_user: { type: 'user_decision', decisionType: 'waiting_user_confirm', ... }
    │   ├─ retry_failed: { type: 'user_decision', decisionType: 'retry_failed', ... }
    │   └─ redecision: 人工重新决策
    │
    └─ 插入到 agent_sub_tasks_step_history
       ├─ interactType: 'response'
       ├─ interactUser: 'human'
       └─ interactNum: nextInteractNum
    │
    ▼
3️⃣ 更新任务状态
    │
    ├─ agent_sub_tasks.status: waiting_user → in_progress
    ├─ updatedAt: 当前时间
    │
    └─ daily_task 更新（如需要）
       └─ 重新决策场景: requiresIntervention = false
    │
    ▼
4️⃣ 触发任务继续执行
    │
    └─ 调用 manuallyExecuteInProgressSubtasks()
       │
       ├─ 1. 实例化 SubtaskExecutionEngine
       └─ 2. 调用 engine.execute()
    │
    ▼
5️⃣ 执行引擎正常处理
    │
    └─ executeAgentBDecisionAndMcp()
       │
       └─ Agent B 根据用户反馈重新决策
          │
          ├─ COMPLETE → completed ✅
          ├─ NEED_USER → waiting_user ⏸️ (再次需要用户)
          ├─ FAILED → failed ❌
          ├─ EXECUTE_MCP → 继续循环 🔄
          └─ ⭐ RE_EXECUTE → pending (让执行Agent重新执行！) ← 新增
```

#### 关键文件位置

| 文件 | 用途 |
|------|------|
| `src/app/api/agents/user-decision/route.ts` | 用户反馈API入口 |
| `src/lib/cron/index.ts` | `manuallyExecuteInProgressSubtasks()` 函数 |
| `src/lib/services/subtask-execution-engine.ts` | 执行引擎核心逻辑 |

## 📊 三、完整执行流程详解

### 3.1 第一阶段：控制器调度

```
┌─────────────────────────────────────────────────────────────────────────┐
│                      阶段 1：控制器调度任务                               │
└─────────────────────────────────────────────────────────────────────────┘

0️⃣ 控制器启动定时扫描
    │
    ├─ 1. 从 agent_sub_tasks 查询待执行任务
    │   └─ 条件：status IN (pending, in_progress, pre_completed, pre_need_support, waiting_user)
    │   └─ 条件：executionDate ≤ 今天
    │
    ├─ 2. 按 fromParentsExecutor + executionDate 分组
    │
    └─ 3. 逐个处理分组
        │
        ├─ 按 orderIndex 排序
        │
        └─ 逐个步骤处理
            │
            ├─ 检查是否全部完成 → 跳过
            ├─ 检查是否有进行中 → 超时检测
            ├─ 检查是否需要支持 → 暂停
            ├─ 检查是否等待用户 → 暂停
            ├─ 检查 pre_completed → Agent B 评审
            ├─ 检查 pre_need_support → Agent B 评审
            └─ 否则 → 执行当前步骤
```

### 3.2 第二阶段：执行Agent能力边界判定

```
┌─────────────────────────────────────────────────────────────────────────┐
│              阶段 2：执行Agent能力边界判定（Executor Agent Workflow）      │
└─────────────────────────────────────────────────────────────────────────┘

1️⃣ 控制器：pending → in_progress
    │
    ├─ 更新 agent_sub_tasks.status = 'in_progress'
    ├─ 设置 startedAt = 当前时间
    │
    ↓
2️⃣ 执行Agent：能力边界判定
    │
    ├─ 调用 ExecutorCapabilityChecker.check()
    ├─ 获取前序任务结果（链式传递）
    │
    ↓
3️⃣ 分支判断
    │
    ├─ [isNeedMcp = false AND isTaskDown = true]
    │   │
    │   ├─ 执行Agent直接执行任务
    │   │   └─ callExecutorAgentDirectly()
    │   │
    │   ├─ 保存 executionResult
    │   │
    │   └─ status → pre_completed ✅
    │
    └─ [其他情况]
        │
        ├─ 保存 executionResult（包含 problem, capabilityType）
        │
        └─ status → pre_need_support ⚠️
```

### 3.3 第三阶段：Agent B 评审（三层验证 + 智能决策）

```
┌─────────────────────────────────────────────────────────────────────────┐
│          阶段 3：Agent B 评审（pre_completed / pre_need_support）        │
└─────────────────────────────────────────────────────────────────────────┘

═══════════════════════════════════════════════════════════════════════════
📌 场景 A：pre_completed（执行Agent说搞定了）
═══════════════════════════════════════════════════════════════════════════

🔴 第一层检查：验证 execution_result 是否真的有内容！
    │
    ├─ 检查：executionResult 是否为 null/空/"null"/""
    │
    ├─ [失败] → 状态改回 pending，让执行Agent重新执行
    │
    └─ [通过] → 继续第二层

🟡 第二层检查：内容长度检查
    │
    ├─ 解析 executionResult，计算内容长度
    │
    ├─ [长度 < 20字符] → 状态改回 pending
    │
    └─ [通过] → 继续第三层

🟢 第三层检查：关键词匹配检查（基本相符性）
    │
    ├─ 从任务标题和描述中提取关键词
    ├─ 检查结果中是否包含至少一个关键词
    │
    ├─ [无匹配] → 状态改回 pending
    │
    └─ [通过] → 进入正常评审流程

✅ 所有验证通过 → executeAgentBDecisionAndMcp()

═══════════════════════════════════════════════════════════════════════════
📌 场景 B：pre_need_support（执行Agent需要帮助）
═══════════════════════════════════════════════════════════════════════════

直接进入 → executeAgentBDecisionAndMcp()
```

### 3.4 第四阶段：Agent B 决策 + MCP 调用（核心循环）

```
┌─────────────────────────────────────────────────────────────────────────┐
│      阶段 4：Agent B 决策 + MCP 调用（executeAgentBDecisionAndMcp）      │
└─────────────────────────────────────────────────────────────────────────┘

1️⃣ 初始化执行上下文
    │
    ├─ 从 task.executionResult 解析 executorResult
    ├─ 查询 capability_list（按 capabilityType）
    │
    ↓
2️⃣ Agent B 主循环（最多 5 次）
    │
    ┌─────────────────────────────────────────────────────────────┐
    │  循环：currentIteration = 1 → 5                              │
    └─────────────────────────────────────────────────────────────┘
    │
    ├─ 构建执行上下文
    │   ├─ executorFeedback（原始任务、问题、建议方案）
    │   ├─ mcpExecutionHistory（MCP执行历史）
    │   ├─ userFeedback（用户反馈）
    │   ├─ taskMeta（任务元数据）
    │   └─ availableCapabilities（可用能力）
    │
    ├─ 调用 Agent B 决策
    │   └─ callAgentBWithDecision()
    │
    └─ 处理决策类型
        │
        ├──────────────────┬──────────────────┬──────────────────┐
        ↓                  ↓                  ↓                  ↓
    COMPLETE           NEED_USER          FAILED          EXECUTE_MCP
        │                  │                  │                  │
        ↓                  ↓                  ↓                  ↓
    结束流程          结束流程          结束流程          继续循环
                                              ↑                  │
                                              └──────────────────┘
                                                   [MCP执行成功/失败后]
```

### 3.5 第五阶段：5 种决策类型详细处理

```
┌─────────────────────────────────────────────────────────────────────────┐
│                  阶段 5：5 种决策类型详细处理                              │
└─────────────────────────────────────────────────────────────────────────┘

═══════════════════════════════════════════════════════════════════════════
📌 决策类型 1：COMPLETE（任务完成）
═══════════════════════════════════════════════════════════════════════════

handleCompleteDecision()
    │
    ├─ 1. 创建交互记录（response）
    │   └─ interactType = 'response'
    │   └─ interactNum = iteration
    │   └─ 包含：mcp_attempts, user_interactions, execution_summary
    │
    ├─ 2. 标记任务完成
    │   └─ markTaskCompleted()
    │   └─ status = 'completed'
    │   └─ completedAt = 当前时间
    │
    └─ 3. return ← 结束整个流程

═══════════════════════════════════════════════════════════════════════════
📌 决策类型 2：NEED_USER（需要用户介入）
═══════════════════════════════════════════════════════════════════════════

handleNeedUserDecision()
    │
    ├─ 1. 创建交互记录 2（response）
    │   └─ interactType = 'response'
    │   └─ interactNum = iteration
    │   └─ 内容：简化，只说"需要用户介入"
    │
    ├─ 2. 创建交互记录 3（request）← 关键！
    │   └─ interactType = 'request'
    │   └─ interactNum = iteration + 1
    │   └─ 内容：pending_key_fields, available_solutions, prompt_message
    │
    ├─ 3. 更新任务状态
    │   └─ status = 'waiting_user'
    │
    └─ 4. return ← 结束流程，等待用户反馈

═══════════════════════════════════════════════════════════════════════════
📌 决策类型 3：FAILED（任务失败）
═══════════════════════════════════════════════════════════════════════════

handleFailedDecision()
    │
    ├─ 1. 创建交互记录（response）
    │   └─ 包含：failed_details
    │
    ├─ 2. 标记等待用户
    │   └─ markTaskWaitingUser()
    │   └─ status = 'waiting_user'
    │
    └─ 3. return ← 结束流程

═══════════════════════════════════════════════════════════════════════════
📌 决策类型 4：EXECUTE_MCP（执行MCP）
═══════════════════════════════════════════════════════════════════════════

executeMcpWithRetry()
    │
    ├─ MCP 尝试循环（最多 3 次）
    │   │
    │   ├─ 1. 执行 MCP
    │   │   └─ executeCapabilityWithParams()
    │   │   └─ genericMCPCall(toolName, actionName, params)
    │   │
    │   ├─ 2. 记录 MCP 尝试
    │   │   └─ 添加到 mcpExecutionHistory
    │   │   └─ 包含：decision, params, result, failureAnalysis
    │   │
    │   ├─ [成功] → return true，继续下一轮决策
    │   │
    │   └─ [失败]
    │       │
    │       ├─ [还有尝试次数]
    │       │   └─ 调用 Agent B 重新决策
    │       │   └─ 获取新的 MCP 参数
    │       │   └─ 继续尝试
    │       │
    │       └─ [无尝试次数] → return false，继续下一轮决策
    │
    └─ 返回是否成功 → Agent B 在下一轮决策中处理

═══════════════════════════════════════════════════════════════════════════
📌 决策类型 5：RE_EXECUTE（让执行Agent重新执行）⭐ 新增
═══════════════════════════════════════════════════════════════════════════

handleReExecuteDecision()
    │
    ├─ 1. 创建交互记录（response）
    │   └─ interactType = 'response'
    │   └─ interactNum = iteration
    │   └─ 包含：re_execute_instruction（给执行Agent的重新执行指令）
    │
    ├─ 2. 更新任务状态
    │   └─ status = 'pending' ← 关键！改为 pending！
    │   └─ updatedAt = 当前时间
    │   └─ executionResult = 重新执行指令（可选，包含用户反馈和Agent B的指示）
    │
    ├─ 3. 清除可能残留状态
    │   └─ startedAt = null（可选，如果需要
    │
    └─ 4. return ← 结束当前流程
              ↓
         等待控制器下次扫描时，
         执行Agent将重新执行！

═══════════════════════════════════════════════════════════════════════════

📌 pre_need_support 状态修改说明：CANNOT_HELP → waiting_user（用户支持）
═══════════════════════════════════════════════════════════════════════════

修改前：
pre_need_support
    │
    └─ [Agent B评审：CANNOT_HELP]
         ↓
      failed ❌

修改后：
pre_need_support
    │
    └─ [Agent B评审：CANNOT_HELP]
         ↓
      waiting_user ⏸️（改为用户支持！

原因：
- Agent B也帮不了 ≠ 任务失败
- 应该转由用户来决定如何处理
- 用户可以选择：
  - 提供更多信息
  - 修改任务要求
  - 取消任务
  - 其他方式
```

## 🔧 四、核心方法架构

### 4.1 公共方法提取

| 方法名 | 用途 | 调用位置 |
|--------|------|----------|
| `getDefaultAccountId()` | 根据事业部选择 accountId | `callAgentB()`, `callAgentBWithDecision()` |
| `buildCapabilitiesText()` | 构建能力清单文本 | `callAgentB()`, `callAgentBWithDecision()` |
| `buildMcpHistoryText()` | 构建MCP执行历史文本 | `callAgentBWithDecision()` |

| `buildExecutionContext()` | 构建完整执行上下文 | `executeDecisionLoop()` |

### 4.2 核心流程方法

```
SubtaskExecutionEngine
│
├─ execute()                          # 主入口
│   ├─ getPendingTasks()              # 获取待执行任务
│   ├─ groupTasks()                    # 分组任务
│   └─ processGroup()                  # 处理分组
│       └─ executeStepTasks()          # 执行步骤任务
│
├─ executeExecutorAgentWorkflow()      # 执行Agent工作流
│   ├─ callExecutorAgent()             # 能力边界判定
```

## 🎯 五、关键决策点总结

### 5.1 执行Agent决策点

| 决策点 | 判断依据 | 影响 |
|---------|----------|------|
| `isNeedMcp` | 是否需要MCP能力协助 | true → pre_need_support, false → 继续判断 |
| `isTaskDown` | 任务是否已完成 | true + !isNeedMcp → pre_completed |
| `capabilityType` | 需要的能力类型 | 用于查询 capability_list |

### 5.2 Agent B决策点

| 决策点 | 可选值 | 影响 |
|---------|---------|------|
| `type` | COMPLETE | 任务完成，结束流程 |
| | NEED_USER | 需要用户介入，等待反馈 |
| | FAILED | 任务失败，转人工 |
| | EXECUTE_MCP | 执行MCP，继续循环 |
| `reasonCode` | MCP_CONTINUE / MCP_RETRY / MCP_NEXT_STEP | EXECUTE_MCP 的细分原因 |
| | TASK_DONE / NO_MCP_NEEDED | COMPLETE 的细分原因 |
| | USER_CONFIRM / USER_SELECT / USER_INPUT | NEED_USER 的细分原因 |
| | MAX_RETRY_EXCEEDED / MCP_ERROR_UNRECOVERABLE | FAILED 的细分原因 |

### 5.3 MCP执行策略

| 策略 | 触发场景 | 说明 |
|------|----------|------|
| `initial` | 首次尝试 | 第一次执行该MCP |
| `retry` | 重试相同方法 | 失败但可重试，使用相同toolName/actionName |
| `switch_type` | 切换方法 | 失败，切换到不同的toolName/actionName |
| `degrade` | 降级方案 | 多次失败，使用备选方案 |

## 📝 六、交互记录结构

### 6.1 交互类型

| `interactType` | 用途 | 成对关系 |
|---------------|------|----------|
| `request` | 发起方提问 | 与下一个 response 成对 |
| `response` | 回答方回复 | 与上一个 request 成对 |
| `agent_consult` | Agent 咨询 | - |
| `agent_response` | Agent 回复 | - |

### 6.2 完整交互序列示例

```
场景：执行Agent需要帮助 → Agent B决策需要用户 → 用户反馈 → Agent B决策执行MCP → 完成

1. request (interactNum=1)
   └─ 执行Agent → 控制器
   └─ 内容：执行Agent的问题

2. response (interactNum=1)
   └─ 控制器 → 执行Agent
   └─ 内容：收到，转给Agent B

3. request (interactNum=2)
   └─ 控制器 → Agent B
   └─ 内容：执行Agent问题 + capability清单

4. response (interactNum=2)
   └─ Agent B → 控制器
   └─ 内容：决策 type = NEED_USER

5. request (interactNum=3)
   └─ Agent B → 用户
   └─ 内容：pending_key_fields, available_solutions

6. [用户反馈]

7. request (interactNum=4)
   └─ 控制器 → Agent B
   └─ 内容：包含用户反馈

8. response (interactNum=4)
   └─ Agent B → 控制器
   └─ 内容：决策 type = EXECUTE_MCP

9. [MCP执行]

10. response (interactNum=5)
    └─ Agent B → 控制器
    └─ 内容：决策 type = COMPLETE
```

## 🎉 九、总结

### 9.1 架构优势

1. **职责清晰**：控制器、执行Agent、Agent B各司其职
2. **可扩展**：新增MCP能力只需更新capability_list
3. **可追溯**：完整的交互历史记录
4. **容错性强**：多层验证、重试机制、降级逻辑
5. **用户友好**：NEED_USER 决策类型支持灵活的用户交互

### 9.2 核心创新点

1. **执行Agent能力边界自判定**：执行Agent自主判断是否需要MCP
2. **Agent B三层验证**：对pre_completed的严格验证防止虚假完成
3. **4种决策类型**：COMPLETE/NEED_USER/FAILED/EXECUTE_MCP覆盖所有场景
4. **MCP重试策略**：initial/retry/switch_type/degrade灵活应对失败
5. **完整交互历史**：request/response成对记录，可追溯可审计
