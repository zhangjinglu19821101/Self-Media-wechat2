### 1. agent-llm.ts 任务拆分提示词优化 ⭐ 推荐（2026-04-07 新增）

**问题描述：**

`agent-llm.ts` 作为任务拆分提示词，对业务理解不够准确，导致任务分配错误。

**具体表现：**
- 示例模板中将"合规校验"任务错误分配给 `insurance-d`
- 实际执行时，insurance-d 反馈"身份不匹配"，需要 Agent B 纠正并切换到 Agent T
- 每次执行都要走"错误分配 → 拒绝 → 切换 → 重新执行"的流程

**影响：**
- 浪费 token（insurance-d 先执行一次，再由 Agent T 执行）
- 浪费执行时间（额外的切换流程）
- 增加系统复杂度

**优化方向：**
1. 重新设计示例模板，确保每个任务类型的 executor 分配正确
2. 明确各 Agent 的职责边界，让任务拆分时就能正确分配
3. 增加"合规校验"等 MCP 任务的正确示例

**预期收益：**
- 减少 50%+ 的无效 token 消耗（大规模使用时成本节约显著）
- 减少执行时间，提升用户体验
- 降低系统复杂度，减少出错概率

**相关文件：**
- `src/lib/agent-llm.ts` - 任务拆分提示词（主要修改）
- `src/lib/agents/prompts/insurance-d-v2.md` - insurance-d 执行提示词（已强化能力边界）
- `src/lib/agents/prompts/agent-b-business-controller.ts` - Agent B 决策提示词

### 3. Agent B 动态加载执行 Agent 提示词与路由能力优化

### 3.1 问题描述

当前 Agent B 的执行 Agent 路由能力存在以下问题：

1. **提示词硬编码**：Agent B 的路由提示词中，执行 Agent 的能力描述是硬编码的
2. **扩展性差**：新增执行 Agent 需要修改 Agent B 的提示词
3. **无法动态感知**：Agent B 无法动态感知执行 Agent 的能力变化

### 3.2 优化方案

#### 方案概述

将执行 Agent 的能力描述从 Agent B 提示词中剥离，改为：

1. **执行 Agent 自描述**：每个执行 Agent 在自己的提示词中定义能力边界
2. **动态加载**：Agent B 在决策时动态加载执行 Agent 的能力描述
3. **统一路由接口**：建立标准的执行 Agent 注册和发现机制

#### 方案细节

### 4. 避免前序任务问题传递的优化需求

### 4.1 问题描述

当前系统存在前序任务问题向后传递的情况：

1. **问题链式传递**：order_index=1 的执行问题可能导致 order_index=2 无法正常执行
2. **错误信息不清晰**：后续任务不知道前序任务为什么失败
3. **无法优雅降级**：系统缺乏对前序失败的处理机制

### 4.2 具体场景

```
order_index=1（文章创作）
  └─ 执行失败（无输出）
      ↓
order_index=2（合规校验）
  └─ 无法执行（没有文章内容）
      ↓
order_index=3（文章修改）
  └─ 无法执行（没有文章内容）
```

### 4.3 优化方案

#### 方案 1：前序任务状态检查机制

在执行当前任务前，检查前序任务是否成功：

```typescript
async checkPrerequisiteTasks(task: SubTask): Promise<{
  canExecute: boolean;
  reason?: string;
  missingInputs?: string[];
}> {
  const prerequisiteTasks = await this.getPrerequisiteTasks(task);
  
  for (const preTask of prerequisiteTasks) {
    if (preTask.status !== 'completed') {
      return {
        canExecute: false,
        reason: `前序任务 order_index=${preTask.orderIndex} 未完成`,
        missingInputs: [preTask.taskTitle]
      };
    }
    
    if (!preTask.resultText && !preTask.resultData) {
      return {
        canExecute: false,
        reason: `前序任务 order_index=${preTask.orderIndex} 无有效输出`,
        missingInputs: [preTask.taskTitle]
      };
    }
  }
  
  return { canExecute: true };
}
```

#### 方案 2：优雅降级机制

当前序任务失败时，提供替代方案：

```typescript
async handlePrerequisiteFailure(task: SubTask, failure: PrerequisiteFailure): Promise<void> {
  // 1. 标记当前任务为"等待前置条件"
  await this.updateTaskStatus(task.id, 'waiting_prerequisite');
  
  // 2. 记录失败原因
  await this.recordTaskIssue(task.id, {
    type: 'PREREQUISITE_FAILURE',
    reason: failure.reason,
    missingInputs: failure.missingInputs
  });
  
  // 3. 尝试恢复或跳过
  if (this.canRecover(failure)) {
    await this.attemptRecovery(task, failure);
  } else {
    await this.skipOrNotifyUser(task, failure);
  }
}
```

#### 方案 3：前序任务输出验证

在任务完成后，验证输出是否有效：

```typescript
async validateTaskOutput(task: SubTask): Promise<{
  isValid: boolean;
  issues?: string[];
}> {
  const output = task.resultText || task.resultData;
  
  if (!output) {
    return { isValid: false, issues: ['任务无输出'] };
  }
  
  // 根据任务类型验证输出格式
  const validator = this.getOutputValidator(task.taskType);
  return validator.validate(output);
}
```

### 5. Agent B 解析 resultData 失败的问题

### 5.1 问题描述

Agent B 在解析任务的 `resultData` 字段时失败，导致：

1. **Agent B 误判**：任务已成功执行（有 MCP 结果），但 Agent B 认为没有结果
2. **用户被误介入**：不必要的用户决策请求
3. **流程阻塞**：用户需要手动确认本可以自动完成的任务

### 5.2 数据库实际情况

```sql
SELECT result_data FROM agent_sub_tasks WHERE order_index = 2;
```

返回内容（简化）：
```json
"map[isCompleted:true mcpParams:map[... ] result:【执行结论】... suggestion:【建议】...]"
```

问题：`map[...]` 不是有效的 JSON 格式。

### 5.3 修复建议

#### 方案 1：修复数据写入逻辑（推荐）

检查 `resultData` 的写入位置，确保：
1. 不要双重序列化
2. 使用 `as any` 时确保类型正确
3. 数据库字段已经是 JSONB 类型，不需要再序列化

**关键检查点**：
- `recordAgentInteraction` 方法
- `executeAgentTExecutorWorkflow` 方法
- MCP 执行结果保存逻辑

#### 方案 2：修复解析逻辑（治标）

在解析 `resultData` 时增加容错处理：

```typescript
let parsed;
if (typeof task.resultData === 'string') {
  // 处理双重序列化的情况
  const cleaned = task.resultData.replace(/^map\[/, '{').replace(/\]$/, '}');
  // 或者：检测到 map[...] 格式时，直接返回空对象
  if (task.resultData.startsWith('map[')) {
    console.warn('[Agent B] resultData 格式异常（map[...]），跳过解析');
    parsed = {};
  } else {
    parsed = JSON.parse(task.resultData);
  }
} else {
  parsed = task.resultData;
}
```

#### 方案 3：统一数据结构（根本解决）

定义明确的 `ExecutorAgentResult` 接口，确保所有地方都遵循：

```typescript
interface ExecutorAgentResult {
  isCompleted: boolean;
  isNeedMcp: boolean;
  isTaskDown: boolean;
  executorOutput: {
    result: string;
    output: string;
    suggestions: string;
  };
}
```

### 6. 兜底逻辑优化需求

### 6.1 问题描述
当前 `callAgentBWithDecision` 方法中存在多层兜底逻辑，导致代码可读性差、维护困难。

### 6.2 当前兜底逻辑分析

#### 第一层：LLM 调用失败兜底
- **位置**：最外层的 try-catch
- **逻辑**：当 LLM 调用失败时，返回 `NEED_USER` 决策
- **问题**：这是合理的兜底，但需要优化错误信息

#### 第二层：响应解析失败兜底
- **位置**：JSON.parse 失败时的 catch 块
- **逻辑**：
  1. 首先检查是否已有有效的 MCP 执行历史
  2. 如果有，直接返回 `COMPLETE` 决策（利用已有 MCP 结果）
  3. 如果没有，返回 `NEED_USER` 决策
- **问题**：这个兜底逻辑很复杂，需要简化

### 6.3 优化建议

#### 建议 1：兜底逻辑分层清晰化
将兜底逻辑按照优先级分层，每层只负责一个层面的兜底：

```
第一层：LLM 调用失败 → 返回 NEED_USER
第二层：响应解析失败 → 检查 MCP 历史 → 有则用之，无则 NEED_USER
```

#### 建议 2：提取独立的兜底方法
将复杂的兜底逻辑提取成独立的方法，提高可读性：

```typescript
private async handleAgentBFailure(
  task: typeof agentSubTasks.$inferSelect,
  executionContext: ExecutionContext,
  error: any,
  currentIteration?: number
): Promise<AgentBDecision> {
  // 检查 MCP 历史
  // 决定返回什么决策
  // 记录交互
}
```

#### 建议 3：减少嵌套层级
当前代码嵌套太深，建议使用早期返回（early return）模式简化：

```typescript
// 优化前
if (condition1) {
  if (condition2) {
    if (condition3) {
      // ...
    }
  }
}

// 优化后
if (!condition1) return fallback1;
if (!condition2) return fallback2;
if (!condition3) return fallback3;
// 主逻辑
```

### 6.4 优化优先级

- **P0（高优先级）**：移除调试代码，保持代码整洁
- **P1（中优先级）**：提取独立方法，提高可读性
- **P2（低优先级）**：重构兜底逻辑，减少嵌套

**状态：** 待实施

### 3. 代码结构优化需求

### 3.2 优化方案
将方法拆分成多个小方法，每个方法只负责一件事：

```typescript
public async callAgentBWithDecision(
  task: typeof agentSubTasks.$inferSelect,
  executionContext: ExecutionContext,
  capabilities: any[],
  currentIteration?: number
): Promise<AgentBDecision> {
  try {
    const prompt = this.buildAgentBPrompt(task, executionContext, capabilities);
    const response = await this.callLLMForAgentB(prompt);
    const decision = await this.parseAgentBResponse(response, task, executionContext);
    await this.recordAgentInteractionIfNeeded(task, executionContext, decision, currentIteration);
    return decision;
  } catch (error) {
    return this.handleAgentBFailure(task, executionContext, error, currentIteration);
  }
}
```

## 新增优化需求（2026-04-17）

### 11. isCompleted 等关键字段的 null 初始化和 LLM 执行错误检测

#### 11.1 问题描述

当前系统在处理 LLM 返回结果时，存在以下问题：

1. **字段初始化问题**：`isCompleted`、`isTaskDown`、`isNeedMcp` 等影响流程的关键字段没有初始化为 `null`
2. **错误检测缺失**：当 LLM 返回的结果中这些字段为 `null` 或 `undefined` 时，系统无法区分是"LLM 执行失败"还是"正常返回 false"
3. **错误的流程流转**：当 LLM 执行出现问题（如格式错误、解析失败）时，结果可能被误判为"任务未完成"，导致 Agent B 错误地介入评审，而不是直接通知用户

#### 11.2 实际影响

```typescript
// 当前代码示例（问题）
const isCompleted = parsed.isCompleted ?? false;  // 如果 LLM 没返回，默认 false
// 结果：LLM 执行失败被误判为"任务未完成"，Agent B 介入评审

// 优化后代码示例（目标）
const isCompleted = parsed.isCompleted ?? null;   // 如果 LLM 没返回，保持 null
// 结果：检测到 null，判定为 LLM 执行错误，直接通知用户
```

#### 11.3 优化方案

##### 方案 1：关键字段初始化为 null

在解析 LLM 返回结果时，将关键字段初始化为 `null` 而非默认值：

```typescript
// parseExecutorResponse 函数优化
function parseExecutorResponse(rawResponse: string): ExecutorDirectResult {
  const parsed = safeJsonParse(rawResponse);
  
  return {
    // 🔥 关键字段初始化为 null，而非默认值
    isCompleted: parsed?.isCompleted ?? null,      // null 表示 LLM 未返回该字段
    isTaskDown: parsed?.isTaskDown ?? null,        // null 表示 LLM 未返回该字段
    isNeedMcp: parsed?.isNeedMcp ?? null,          // null 表示 LLM 未返回该字段
    
    // 其他字段保持原有逻辑
    result: parsed?.result ?? null,
    output: parsed?.output ?? null,
    suggestion: parsed?.suggestion ?? null,
    needsMcpSupport: parsed?.needsMcpSupport ?? null,
    mcpParams: parsed?.mcpParams ?? null,
  };
}
```

##### 方案 2：LLM 执行错误检测机制

在转换结果为 Agent 结果时，检测关键字段是否为 null：

```typescript
// convertExecutorDirectToAgentResult 函数优化
function convertExecutorDirectToAgentResult(
  directResult: ExecutorDirectResult,
  rawResponse: string
): { result: ExecutorAgentResult; hasLlmError: boolean; errorReason?: string } {
  
  // 🔥 检测 LLM 是否返回了关键字段
  const requiredFields = ['isCompleted', 'result'];
  const missingFields = requiredFields.filter(field => 
    directResult[field as keyof ExecutorDirectResult] === null
  );
  
  if (missingFields.length > 0) {
    // LLM 执行错误，关键字段缺失
    return {
      result: createErrorResult('LLM_EXECUTION_ERROR', `缺失关键字段: ${missingFields.join(', ')}`),
      hasLlmError: true,
      errorReason: `LLM 返回结果格式错误，缺失字段: ${missingFields.join(', ')}`
    };
  }
  
  // 正常处理逻辑...
  return { result: normalResult, hasLlmError: false };
}
```

##### 方案 3：错误处理流程

当检测到 LLM 执行错误时，直接通知用户，不经过 Agent B 评审：

```typescript
// 执行任务后的处理逻辑
async function handleExecutionResult(
  task: SubTask,
  executionResult: { result: ExecutorAgentResult; hasLlmError: boolean; errorReason?: string }
): Promise<void> {
  
  if (executionResult.hasLlmError) {
    // 🔥 LLM 执行错误，直接通知用户，不经过 Agent B
    await this.recordTaskError(task.id, {
      type: 'LLM_EXECUTION_ERROR',
      reason: executionResult.errorReason,
      rawResponse: executionResult.rawResponse
    });
    
    // 直接通知用户，创建用户决策任务
    await this.createUserDecisionTask(task, {
      type: 'LLM_ERROR',
      message: `执行 Agent 返回结果格式错误：${executionResult.errorReason}`,
      options: ['重试', '跳过', '修改指令']
    });
    
    return;  // 不进入 Agent B 评审流程
  }
  
  // 正常流程：Agent B 评审
  await this.proceedToAgentBReview(task, executionResult.result);
}
```
