
# Agent 交互流程设计文档（与代码实现一致版）

## 一、核心流程概览

```
1. 执行Agent（insurance-d）能力边界判定
   ↓
2. 控制器直接查询 capability_list
   ↓
3. Agent B 决策并输出 MCP 调用参数
   ↓
4. 控制器执行 MCP 调用
   ↓
5. 返回结果给执行Agent
   ↓
6. 更新任务状态
```

## 二、执行Agent输出规范（与代码一致）

### 2.1 输出格式（驼峰命名）

```typescript
interface ExecutorAgentResult {
```

## 四、核心流程说明（与代码一致）

### 4.1 流程特点

1. **无 list_capabilities 交互步骤**：控制器直接查询 capability_list，不经过 Agent B 请求
2. **命名统一使用驼峰**：如 `isNeedMcp`, `capabilityType`, `solutionNum`, `isNotifyAgentA`
3. **Agent B 直接输出完整参数**：不区分 `core_params` 和 `default_params`
4. **暂不实现五步校验**：仅做基础字段检查
5. **暂不实现参数合并逻辑**：直接使用 Agent B 输出的参数

### 4.2 详细步骤

| 步骤 | 说明 | 相关代码 |
|------|------|---------|
| 1 | 执行Agent能力边界判定 | `ExecutorCapabilityChecker.check()` |
| 2 | 控制器查询 capability_list | `queryCapabilityList()` |
| 3 | 调用 Agent B 决策 | `callAgentB()` |
| 4 | 执行 MCP 调用 | `executeCapability()` |
| 5 | 返回结果给执行Agent | `sendBackToExecutor()` |
| 6 | 更新任务状态 | `markTaskCompleted()` |

## 六、与原设计文档的主要差异

| 维度 | 原设计文档 | 当前实现（本文档） |
|------|-----------|-------------------|
| 命名规范 | 下划线（`is_need_mcp`） | 驼峰（`isNeedMcp`） |
| list_capabilities | 有此交互步骤 | 无此步骤，直接查询 |
| Agent B 输出 | `list_capabilities`, `solution_num` | `solutionNum`, `toolName`, `actionName`, `params` |
| 参数分离 | 区分 `core_params` 和 `default_params` | 不区分，直接传完整参数 |
| 五步校验 | 有详细五步校验 | 仅基础检查 |
