# Agent 指令传达验证机制

## 📋 概述

Agent A（总裁）在向其他 Agent 下达指令后，必须验证指令是否成功传达，并持续监控执行进度。本文档定义了验证机制和检查规则。

## 🎯 验证机制

### 1. 实时指令验证

**API**：`POST /api/agents/send-command`

**验证方式**：
- ✅ 流式响应：对方 Agent 会立即响应
- ✅ 内容确认：检查响应内容，确认对方理解指令
- ✅ 状态确认：确认对方表示"已收到"、"明白"、"开始执行"

**验证成功标志**：
```
✅ "收到，明白" / "好的，我会执行" / "明白了，开始执行"
✅ 详细说明执行计划
✅ 提出相关问题需要澄清
```

**验证失败标志**：
```
❌ 没有响应
❌ 响应内容异常（乱码、错误信息）
❌ 对方表示无法理解指令
❌ 网络错误或 API 错误
```

**失败处理**：
1. 重试最多3次
2. 如果仍失败，通知用户
3. 记录失败原因到记忆系统

### 2. 任务管理验证

**API**：`POST /api/agents/tasks`

**验证方式**：
- ✅ 任务ID：创建任务后返回 task_id
- ✅ 状态跟踪：定期查询任务状态
- ✅ 反馈查询：查看任务执行结果

**验证步骤**：

#### Step 1: 记录任务ID
```
创建任务后，返回：
{
  "success": true,
  "data": {
    "task": {
      "taskId": "task-1234567890-abc123",
      "status": "pending",
      ...
    }
  }
}

将 task_id 记录到记忆系统
```

#### Step 2: 定期检查状态（每10分钟）
```bash
GET /api/agents/A/tasks?status=in_progress
```

**检查要点**：
- 任务状态是否从 pending 变为 in_progress
- 任务状态是否变为 completed
- 任务状态是否变为 failed

#### Step 3: 查看任务反馈
```bash
GET /api/agents/A/feedback
```

**反馈内容**：
```json
{
  "success": true,
  "data": {
    "feedbacks": [
      {
        "taskId": "task-1234567890-abc123",
        "fromAgent": "B",
        "command": "调研AI内容检测工具",
        "result": "调研报告内容...",
        "completedAt": "2025-02-10T10:00:00Z"
      }
    ]
  }
}
```

## 🔍 任务状态检查规则

### 状态流转

```
pending（待执行）
    ↓
in_progress（执行中）
    ↓
completed（已完成）/ failed（已失败）
```

### 检查频率

| 任务类型 | 预计时间 | 检查频率 |
|---------|---------|---------|
| 简单任务 | < 10分钟 | 立即检查 |
| 中等任务 | 10-60分钟 | 每10分钟 |
| 复杂任务 | > 60分钟 | 每30分钟 |

### 异常处理

**长时间未开始**：
```
条件：任务状态为 pending，创建时间 > 1小时
处理：
1. 通知用户：任务可能未被接收
2. 催促对方 Agent
3. 或调整任务优先级
```

**任务失败**：
```
条件：任务状态为 failed
处理：
1. 查看失败原因（result 字段）
2. 分析失败原因
3. 向用户报告并提供建议
4. 记录失败经验
```

**任务超期**：
```
条件：超过 deadline 未完成
处理：
1. 催促对方 Agent
2. 评估是否需要调整 deadline
3. 通知用户情况
```

## 📊 反馈分析规则

### 评估标准

**✅ 成功**：
- 反馈内容完整
- 达到预期目标
- 质量符合要求
- 按时完成

**⚠️ 部分成功**：
- 反馈内容基本完成
- 但需要改进
- 或有小部分未完成

**❌ 失败**：
- 未完成指令要求
- 质量不达标
- 或方向完全错误

**🔄 需要调整**：
- 理解错误，方向不对
- 需要重新下达指令
- 或补充说明

### 评估后处理

**成功**：
```
1. 向用户报告任务完成
2. 总结成功经验
3. 记录到记忆系统
4. 评估执行质量，记录优秀案例
```

**部分成功**：
```
1. 向用户报告部分完成
2. 说明未完成部分
3. 要求对方改进
4. 或补充下达指令
```

**失败**：
```
1. 向用户报告任务失败
2. 分析失败原因
3. 提出解决方案
4. 决定是否重新下达
```

**需要调整**：
```
1. 向用户报告理解错误
2. 重新明确指令要求
3. 下达更详细的指令
4. 确保对方理解
```

## 🎯 实际使用案例

### 案例 1：实时指令验证

**场景**：让 Agent B 检查系统状态

```javascript
// Agent A 下达指令
POST /api/agents/send-command
{
  "fromAgentId": "A",
  "toAgentId": "B",
  "command": "请检查系统运行状态，看看是否有异常",
  "commandType": "instruction",
  "priority": "high"
}

// Agent B 流式响应（流式内容）
data: {"content":"收到，正在检查系统状态..."}
data: {"content":"\n\n系统状态检查完成："}
data: {"content":"\n- CPU使用率：45%"}
data: {"content":"\n- 内存使用率：60%"}
data: {"content":"\n- 磁盘使用率：70%"}
data: {"content":"\n\n没有发现异常。"}
data: [DONE]

// Agent A 验证
✅ 对方立即响应
✅ 确认收到指令
✅ 执行了检查
✅ 反馈了结果
→ 验证成功
```

## 🔄 完整验证流程

### 实时指令流程

```
Agent A 下达指令
    ↓
立即查看流式响应
    ↓
对方是否响应？
├─ 否 → 重试最多3次 → 仍失败 → 通知用户
└─ 是 → 继续检查

对方是否理解指令？
├─ 否 → 解答疑问，重新说明
└─ 是 → 验证成功

对方是否开始执行？
├─ 否 → 询问原因，提供帮助
└─ 是 → 监控执行

执行完成，收到结果
    ↓
评估执行质量
    ↓
记录到记忆系统
    ↓
向用户报告结果
```

### 任务管理流程

```
Agent A 创建任务
    ↓
记录 task_id
    ↓
保存到记忆系统
    ↓
任务进入待执行队列（pending）
    ↓
每10分钟检查状态
    ↓
状态变为 in_progress？
├─ 否，超过1小时 → 通知用户，催促对方
└─ 是 → 继续监控

状态变为 completed/failed？
├─ 否，继续检查
└─ 是 → 查看反馈结果

查看反馈内容
    ↓
评估执行质量
├─ 成功 → 向用户报告，记录经验
├─ 部分成功 → 要求改进，或补充指令
├─ 失败 → 分析原因，提供建议
└─ 需要调整 → 重新下达指令
    ↓
记录到记忆系统
    ↓
向用户报告结果
```

## 📌 重要提示

1. **主动检查**：不要等待对方主动反馈，要主动检查
2. **定期检查**：按照频率定期检查任务状态
3. **及时处理**：发现异常及时处理，不要拖延
4. **记录经验**：所有验证结果和经验都要记录
5. **通知用户**：重要情况及时向用户报告

## 🎓 总结

| 指令类型 | 验证方式 | 检查频率 | 反馈方式 |
|---------|---------|---------|---------|
| 实时指令 | 流式响应 | 立即 | 实时反馈 |
| 任务管理 | 状态查询 | 每10-30分钟 | 任务反馈 |

**核心原则**：
- ✅ 主动验证，不要被动等待
- ✅ 定期检查，掌握最新状态
- ✅ 及时处理，避免问题积累
- ✅ 记录经验，持续优化策略

通过遵循这些验证机制，Agent A 可以确保指令成功传达并有效监控执行！
