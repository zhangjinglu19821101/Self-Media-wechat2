## 一、阻断性问题 (Blockers) — 必须修复，否则无法编译/运行

### B1. Context 导入了不存在于 types.ts 的常量

**文件**: `creation-guide-context.tsx` 第22行

```typescript
import { 
  USER_DEFAULT_7_SECTION_STRUCTURE    // 此常量已从 types.ts 移至 structure-templates.ts
} from './types';
```

**问题**: 上轮优化将 `USER_DEFAULT_7_SECTION_STRUCTURE` 从 `types.ts` 迁移到了 `structure-templates.ts`，但 Context 文件仍从 `./types` 导入。**编译失败**。

**修复**: 改为从 `structure-templates.ts` 导入。

---

### B2. `import type` 导入了运行时值

**文件**: `core-anchor-input-optimized.tsx` 第11-16行

```typescript
import type {
  CoreAnchorData,
  CoreAnchorInputProps,
  ValidationResult,
  WORD_COUNT_RANGES       // 这是 const 值，不是类型
} from './types';
```

**问题**: `WORD_COUNT_RANGES` 是运行时常量 (`as const`)，不是类型。`import type` 编译后被擦除，导致运行时 `WORD_COUNT_RANGES` 为 `undefined`，所有字数验证崩溃。

**修复**: 拆分为两条 import：
```typescript
import type { CoreAnchorData, CoreAnchorInputProps, ValidationResult } from './types';
import { WORD_COUNT_RANGES } from './types';
```

---

### B3. `useState` 未导入

**文件**: `use-debounced-storage.ts` 第3行 vs 第20-21行

```typescript
import { useCallback, useEffect, useRef, useMemo } from 'react';  // 缺少 useState

const [isSaving, setIsSaving] = useState(false);     // 未导入
const [lastSaved, setLastSaved] = useState<Date | null>(null);  // 未导入
```

**问题**: 编译直接报错。

**修复**: 添加 `useState` 到 import。

---

### B4. `loadFromStorage` 缺少必需参数

**文件**: `page-v3.tsx` 第254行

```typescript
const saved = loadFromStorage(STORAGE_KEY);  // 缺少第二个参数 defaultValue
```

函数签名要求两个参数：
```typescript
export function loadFromStorage<T extends object>(key: string, defaultValue: T): T
```

**问题**: TypeScript 编译错误。

**修复**: 补充默认值参数。

---

### B5. 新旧类型系统并行冲突

**矛盾对比**:

| 属性 | 旧版 (structure-selector.tsx) | 新版 (types.ts + structure-templates.ts) |
|------|------|------|
| Section标识 | `order: number` | `id: string` |
| Section名称 | `title: string` | `name: string` |
| Section字数 | `suggestedLength: string` | `suggestedWordCount: number` |
| Template标记 | `isDefault: boolean` | `isFixed?: boolean` + `isUserExclusive?: boolean` |

**影响**: `page-v3.tsx` 从 Context（新版类型）获取 `selectedStructure`，传给 `StructureSelector`（旧版类型），运行时访问 `section.order` 返回 `undefined`，页面崩溃。

同理 `material-provider.tsx`、`creation-controller.tsx` 各自定义了重复类型，极易分叉。

**修复**: 统一所有组件从 `types.ts` 导入类型，删除组件内重复定义，更新组件逻辑适配新类型。

## 二、严重问题 (Critical) — 逻辑缺陷，影响正确性

### C1. Selector Hooks 是假的 — 无法隔离重渲染

**文件**: `creation-guide-context.tsx` 第300-349行

```typescript
export function useCoreAnchorData() {
  const { state, updateCoreAnchor } = useCreationGuide();  // 订阅了整个 Context
  return { data: state.coreAnchorData, update: updateCoreAnchor };
}
```

**问题**: 这些 Hook 被标注为"Selector Hooks (性能优化)"，但它们调用了 `useCreationGuide()`，即订阅了 **整个 Context value**。当 Context 中任意字段变化（如 `activeTab` 变化），**所有**使用了 Selector Hook 的组件都会重渲染。

当前实现与直接用 `useCreationGuide()` **没有任何性能差别**，注释具有误导性。

**修复方案**（三选一）:
1. **推荐**: 使用 `use-context-selector` 库实现真正的 Selector
2. **简单**: 拆分为多个独立 Context
3. **妥协**: 删除"性能优化"注释，明确标注只是便利方法

---

### C2. useReducer 不兼容异步 initialState

**文件**: `creation-guide-context.tsx` 第199-204行

```typescript
const mergedInitialState = useMemo(() => ({
  ...INITIAL_STATE, ...initialState,
}), [initialState]);

const [state, dispatch] = useReducer(creationGuideReducer, mergedInitialState);
```

**问题**: `useReducer` 的初始状态只在 **首次挂载时** 使用。`page-v3.tsx` 中 `initialState` 是异步从 localStorage 加载的，加载完成后 Provider 已用 `INITIAL_STATE` 初始化，`mergedInitialState` 的变化不会反映到 reducer 实际状态中。

**修复**: 使用 `useReducer` 惰性初始化（第三参数），或在 `initialState` 变化时 dispatch `LOAD_STATE`。

---

### C3. sanitizeTemplateInput 编码顺序错误

**文件**: `safety-utils.ts` 第27-36行

```typescript
return text
  .replace(/</g, '&lt;')    // 第1步: < 变成 &lt;
  .replace(/>/g, '&gt;')    // 第2步: > 变成 &gt;
  .replace(/&/g, '&amp;')   // 第3步: & 变成 &amp; — 此时将第1步产生的 &lt; 变成 &amp;lt;
```

**问题**: HTML实体编码必须 **先编码 `&`**，再编码其他字符。当前顺序导致双重编码。

示例: 输入 `<script>`
- 当前结果: `&amp;lt;script&amp;gt;` (双重编码)
- 正确结果: `&lt;script&gt;` (正常转义)

**修复**: `&` 的替换放到最前面。

---

### C4. 同名类型 CreationGuideState 定义冲突

**文件对比**:

| 字段 | types.ts (旧版) | creation-guide-context.tsx (新版) |
|------|------|------|
| loadingState | `LoadingState` | 无 |
| error | `CreationGuideError \| null` | `string \| null` |
| activeTab | 无 | `number` |
| isSaving | 无 | `boolean` |
| lastSaved | 无 | `Date \| null` |

**问题**: TypeScript 的 declaration merging 会将同名 interface 合并，导致两个文件中所有字段都成为必需，几乎必定造成编译错误。

**修复**: 删除 `types.ts` 中的旧版定义。

---

### C5. page-v3.tsx 自动保存为空壳

**文件**: `page-v3.tsx` 第266-269行

```typescript
useEffect(() => {
  // 这里可以集成防抖自动保存Hook
  // 先手动实现简单的保存逻辑
}, [initialState]);
```

**问题**: 自动保存完全未实现。依赖数组 `[initialState]` 只在初始化时触发一次，后续数据修改不会触发保存。用户输入的数据刷新后丢失。

**修复**: 集成 `useDebouncedStorage` 或在 Context 中实现保存逻辑。

## 六、评审总结

### 核心结论

本轮优化的 **设计意图是好的**——Context 集中管理方向正确，结构配置化思路清晰。但 **实现执行层面存在严重问题**：

1. **5个阻断性Bug**: 代码无法通过编译或运行时崩溃
2. **5个严重逻辑缺陷**: 数据丢失、类型冲突、性能假象
3. **新旧代码未对齐**: 优化了新文件，但旧组件文件（structure-selector.tsx、material-provider.tsx、creation-controller.tsx）仍使用旧类型，形成了两套互不兼容的类型系统

### 根因分析

核心根因是 **增量优化时缺乏全局类型一致性校验**：
- 在 `types.ts` 中修改了 `StructureTemplate` 接口（`order` → `id`, `title` → `name`, `suggestedLength` → `suggestedWordCount`）
- 但没有同步更新使用这些类型的组件文件
- 也没有删除旧组件文件中的重复类型定义
- 导致两套类型系统并行运行

### 修复建议

**优先级排序**:
1. **先修阻断性Bug** (B1-B4): 确保代码能编译运行
2. **统一类型系统** (B5): 这是最核心的工作，需要更新所有组件适配新类型
3. **修严重逻辑** (C1-C5): 确保功能正确
4. **清理中等/轻微问题** (M/m): 提升代码质量

**关键建议**: 在完成修复后，增加 `pnpm build` 和 `pnpm ts-check` 作为提交门禁，避免类型不一致的问题再次发生。
