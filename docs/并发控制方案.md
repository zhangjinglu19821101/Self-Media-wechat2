# 🏆 并发控制方案：状态机 + 乐观锁

## 📋 方案概述

本方案采用**业界最佳实践**：状态机 + 乐观锁，用于解决用户反馈API和定时任务之间的并发冲突问题。

---

## 🎯 核心设计原则

### 1️⃣ 用户反馈优先
- **用户操作 > 定时任务**
- 当检测到最近有用户反馈时，定时任务自动让道

### 2️⃣ 严格的状态流转
- 只有预定义的状态转换是允许的
- 防止非法状态跳转

### 3️⃣ 乐观锁并发控制
- 使用版本号机制
- 冲突时自动重试（指数退避）

### 4️⃣ 可追溯性
- 所有状态变更都有记录
- 支持审计和问题排查

---

## 🗂️ 状态定义

```typescript
type TaskStatus = 
  | 'pending'           // 待执行
  | 'in_progress'       // 执行中
  | 'waiting_user'      // 等待用户反馈
  | 'pre_completed'     // 待审核（执行Agent说完成了）
  | 'pre_need_support'  // 待审核（执行Agent需要帮助）
  | 'need_support'      // 需要支持
  | 'completed'         // 已完成
  | 'cancelled';        // 已取消
```

## 🚀 使用指南

### 1️⃣ 基础使用：带乐观锁的状态更新

```typescript
import { taskStateMachine, TaskStatus, OperationSource } from '@/lib/services/task-state-machine';

// 用户反馈场景
const result = await taskStateMachine.updateStateWithLock(
  taskId,                    // 任务ID
  currentVersion,            // 当前版本号（从数据库读取）
  'in_progress' as TaskStatus,  // 新状态
  'user_feedback' as OperationSource,  // 操作来源
  {
    executionResult: null,   // 清除等待状态
    metadata: {
      userFeedbackProcessed: true,
      processedAt: new Date().toISOString(),
    }
  }
);

if (result.success) {
  console.log('状态更新成功，新版本号:', result.newVersion);
} else if (result.conflict) {
  console.log('并发冲突，请重试');
} else if (result.invalidTransition) {
  console.log('非法状态转换');
}
```

### 2️⃣ 高级使用：自动重试

```typescript
// 带自动重试的状态更新（指数退避）
const result = await taskStateMachine.updateStateWithRetry(
  taskId,
  'in_progress' as TaskStatus,
  'user_feedback' as OperationSource,
  { executionResult: null },
  3  // 最多重试3次
);
```

### 3️⃣ 定时任务场景：让道给用户反馈

```typescript
// 定时任务中使用
async function handleTimeoutTask(taskId: string) {
  const result = await taskStateMachine.updateStateWithRetry(
    taskId,
    'waiting_user' as TaskStatus,
    'scheduled_task' as OperationSource,  // 标记为定时任务
    { executionResult: { timeout: true } }
  );

  if (!result.success && result.message?.includes('用户反馈')) {
    // 定时任务检测到用户反馈，自动让道
    console.log('检测到用户反馈，定时任务已放弃执行');
    return;
  }

  // 继续处理...
}
```

## 📊 冲突处理流程

```
1. 查询当前任务状态和版本号
   ↓
2. 检查状态转换是否合法
   ↓
3. 如果是定时任务，检查是否有最近用户反馈
   ├─→ 有用户反馈 → 放弃执行，让道给用户
   └─→ 无用户反馈 → 继续
   ↓
4. 带版本号条件更新
   ├─→ 成功 → 返回新版本号
   └─→ 失败（版本不匹配）→ 冲突，重试
```

## 🎯 最佳实践

### ✅ Do's

1. **总是使用状态机**：不要直接更新 `agent_sub_tasks.status`
2. **正确标记操作来源**：`user_feedback` 或 `scheduled_task`
3. **处理冲突结果**：检查 `result.conflict` 和 `result.invalidTransition`
4. **使用重试机制**：对用户反馈操作使用 `updateStateWithRetry`

### ❌ Don'ts

1. **不要绕过状态机**：直接 `UPDATE agent_sub_tasks SET status = ...`
2. **不要忽略冲突**：收到冲突错误时应该重试或提示用户
3. **不要定义非法转换**：比如从 `completed` 转回 `in_progress`
4. **不要混用操作来源**：定时任务不要标记为 `user_feedback`

## 📈 性能考虑

### 乐观锁 vs 悲观锁

| 特性 | 乐观锁（推荐） | 悲观锁 |
|------|---------------|--------|
| 适用场景 | 读多写少 | 写多读少 |
| 并发性能 | 高 | 低 |
| 实现复杂度 | 中 | 低 |
| 重试机制 | 需要 | 不需要 |

### 本方案优势
- **无锁等待**：不阻塞其他操作
- **高性能**：适合高并发场景
- **简单可靠**：不需要额外的基础设施
