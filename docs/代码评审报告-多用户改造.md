## 二、P0 级问题（必须修复，否则核心流程瘫痪）

### P0-1: 后端内部 API 调用被 401 阻断

**问题描述**:  
`requireAuth()` 依赖 `auth()` 获取 NextAuth session，而后端内部（Agent、Cron、MCP 等）通过 `fetch('http://localhost:5000/api/...')` 调用 API 时不携带 session cookie，导致所有内部调用返回 401。

**受影响的调用链**:

| 调用方 | 目标 API | 影响 |
|--------|---------|------|
| `cron-split-trigger.ts:150` | `/api/agents/agent-b/unified-split-task` | 定时拆解完全失效 |
| `cron-split-trigger.ts:181` | `/api/agents/insurance-a/split-task` | 保险拆解失效 |
| `split-retry-manager.ts:89` | `/api/tasks/[taskId]/split` | 拆解重试失效 |
| `mcp/web-search-executor.ts:33,58,83` | `/api/search` | MCP 搜索能力完全失效 |
| `commands/send/route.ts` | `/api/command-results` + `/api/agents/tasks/*/status` + `/api/agents/tasks/*/result` | 命令发送流程断裂 |
| `commands/reject/route.ts` | `/api/agents/tasks/*/split` | 命令拒绝流程断裂 |
| `agents/[id]/chat/route.ts` | `/api/agents/[id]/commands` | Agent 聊天流程断裂 |

**根因**: `requireAuth(request)` → `auth()` → NextAuth session cookie → 内部请求无 cookie → 返回 null → 401

**修复方案**:
1. `requireAuth()` 增加内部请求识别：检查请求头 `x-internal-call` 或来源 IP 为 `127.0.0.1`
2. 内部请求使用 Service Token 认证，绕过 session 检查
3. 或者：内部调用直接调用 service 层方法而非走 HTTP API

```typescript
// 建议修复方案：requireAuth 增加内部调用白名单
export async function requireAuth(request: NextRequest) {
  // 内部调用检测
  const internalToken = request.headers.get('x-internal-token');
  if (internalToken === process.env.INTERNAL_API_TOKEN) {
    const workspaceId = extractWorkspaceId(request) || 'default-workspace';
    return { accountId: 'system', workspaceId };
  }
  
  // 正常用户认证流程
  const ctx = await getAuthContext(request);
  if (!ctx) {
    return NextResponse.json({ error: '未登录或登录已过期', code: 'AUTH_REQUIRED' }, { status: 401 });
  }
  return ctx;
}
```

### P0-2: 核心数据流 workspaceId 未传递（数据混写风险）

**问题描述**:  
多个 INSERT 操作没有传递 `workspaceId`，导致新创建的数据 `workspace_id` 为 NULL。在多用户环境下，这些数据将无法被任何 workspace 查询到，形成"幽灵数据"。

**受影响的 INSERT 操作**:

| 文件 | 操作 | 缺失字段 |
|------|------|---------|
| `save-split-result-v2.ts` | INSERT daily_task + agent_tasks | `workspace_id` |
| `agent-sub-tasks/confirm-split/route.ts` | INSERT agent_sub_tasks | `workspace_id` |
| `agent-sub-tasks/confirm-split-fix/route.ts` | INSERT agent_sub_tasks (原生SQL) | `workspace_id` |
| `split/confirm/route.ts` | INSERT agent_sub_tasks (原生SQL) | `workspace_id` |
| `agents/[id]/subtasks/route.ts` (两处) | INSERT agent_sub_tasks + daily_task | `workspace_id` |
| `exceptions/[failureId]/resolve/route.ts` | INSERT daily_task | `workspace_id` |
| `subtask-execution-engine.ts` (3处INSERT) | auto-split + 大纲确认 INSERT | `workspace_id` |
| `command-result-service.ts` | createAgentTaskWithDuplicateCheck + createDailyTaskWithDuplicateCheck | `workspace_id` |
| `commands/route.ts` POST | 创建 dailyTask | `workspace_id` |

**根因**: `requireAuth()` 只在 GET/POST 入口处获取了 `workspaceId`，但未将其传递到后续的 service 层 INSERT 操作中。

**修复方案**:  
在每个入口路由中，将 `workspaceId` 传递到 service 层方法，确保所有 INSERT 操作都写入 workspace_id。

### P0-3: `style-deposition-service.ts` INSERT 字段名错误

**问题描述**:  
Schema 中 `styleAssets` 表的字段已从 `userId` 改名为 `workspaceId`，但 `saveDepositionResults()` 方法中仍然使用 `userId` 作为 INSERT 字段名。

**文件**: `src/lib/services/style-deposition-service.ts`  
**行号**: 593, 665

```typescript
// 当前代码（错误）
userId: userId ?? null,

// 应改为
workspaceId: userId ?? null,  // 或 workspaceId: workspaceId ?? null,
```

**影响**: 运行时 Drizzle INSERT 会因字段名不匹配而报错，风格初始化流程完全不可用。

### P0-4: middleware 同样阻断内部调用

**问题描述**:  
`src/middleware.ts` 使用 `auth()` 检查 session，后端内部 fetch 请求无 cookie，会被 middleware 在到达路由处理函数之前就拦截。

**修复方案**:  
middleware 增加内部调用识别逻辑（同 P0-1），或者对携带 `x-internal-token` 的请求放行。

## 三、P1 级问题（功能异常，需修复）

### P1-1: WebSocket 用户认证基于空表

**问题描述**:  
`websocket-auth.ts` 的 `authenticateUser()` 通过查询 `accountSessions` 表验证 token。但 NextAuth 配置使用 `strategy: 'jwt'`，session 信息存储在客户端 JWT cookie 中，**永远不会写入 `accountSessions` 表**。因此该表始终为空，所有用户 WebSocket 连接认证必定失败。

**文件**: `src/lib/websocket-auth.ts:75-85`

**修复方案**:
1. 改用 NextAuth 的 `getToken()` 方法验证 JWT token
2. 或者改用 `session` 方案：前端通过 `getSession()` 获取 session token，WS 传给后端，后端调用 NextAuth 的 `decode()` 验证

```typescript
// 建议修复方案
import { getToken } from 'next-auth/jwt';

export async function authenticateUser(token: string, workspaceId?: string) {
  try {
    // 使用 NextAuth JWT 验证
    const decoded = await getToken({ 
      req: { headers: { cookie: `next-auth.session-token=${token}` } } as any,
      secret: process.env.NEXTAUTH_SECRET 
    });
    if (!decoded?.sub) return null;
    // ... 后续逻辑
  } catch { return null; }
}
```

### P1-2: WorkspaceSwitcher 不触发 workspace-changed 事件

**问题描述**:  
`WorkspaceSwitcher` 切换 workspace 时只做了 `router.refresh()` 和 `localStorage.setItem()`，没有调用 `setCurrentWorkspaceId()` 或 dispatch `workspace-changed` 事件。

**影响**: 
- `api/client.ts` 的 `getCurrentWorkspaceId()` 在下次页面刷新前不会更新
- AppNavbar 等 `workspace-changed` 事件监听器不会触发

**文件**: `src/components/workspace-switcher.tsx`

**修复方案**: 切换时调用 `setCurrentWorkspaceId(id)` 替代直接操作 localStorage。

### P1-3: `style-similarity-validator.ts` 参数语义混乱

**问题描述**:  
`validate()` 方法的第三个参数名为 `userId`，但从 `style/init-from-upload/route.ts` 调用时传入的是 `workspaceId`。由于 `styleTemplateService.listTemplates()` 已改为接收 `workspaceId`，功能上碰巧可能正常，但代码语义严重混乱。

**文件**: `src/lib/services/style-similarity-validator.ts:83`

### P1-4: `agents/B/intervene/route.ts` 重复函数处理不当

**问题描述**:  
该文件原本有两个 `export async function POST`，脚本添加 `requireAuth` 后结构更加混乱。当前将第二个重命名为 `_deprecated_processAnswer`，但它仍然是一个顶级函数，且参数类型使用旧式 `context: { params: { id: string } }` 而非 Next.js 16 的 `Promise<params>` 模式。

**建议**: 完全移除废弃函数，或将其移到单独的 API 路由文件。

## 四、P2 级问题（功能不完整，需后续修复）

### P2-1: 前端页面 fetch 未使用 apiGet/apiPost 封装

**问题描述**:  
前端页面（`digital-assets/page.tsx`、`account-management/page.tsx`、`materials/page.tsx`、`full-home/page.tsx`、`style-init/page.tsx` 等）的 fetch 调用全部未使用 `@/lib/api/client.ts` 封装，不携带 `x-workspace-id` header。

**特别严重**: `account-management/page.tsx:103-104` 仍然使用 `?userId=default-user` 查询参数：
```typescript
fetch('/api/style-templates?userId=default-user'),
fetch('/api/platform-accounts?userId=default-user'),
```

**影响**: 前端操作无法正确传递 workspaceId，多 workspace 数据隔离在前端层面完全失效。

### P2-2: `digital-asset-service.ts` 方法签名混合 userId/workspaceId

**问题描述**:  
`createStyleRule()` 和 `listStyleRules()` 同时接受 `userId` 和 `workspaceId` 参数，内部用 `workspaceId || userId` 选择。这种向后兼容的方式虽然不会导致运行时错误，但长期维护会越来越混乱。

**建议**: 统一使用 `workspaceId`，`userId` 标记为 `@deprecated`。

### P2-3: WorkspaceSwitcher 加载 workspaces 时不带认证 header

**问题描述**:  
`loadWorkspaces()` 调用 `/api/workspaces` 时不携带 session cookie 或 `x-workspace-id` header。虽然 NextAuth 的 session cookie 会自动携带（浏览器行为），但如果 API 需要额外的 workspace header 才能工作，就会出问题。

### P2-4: `prompt-assembler-service.ts` 未改造

**问题描述**:  
`PromptAssemblyOptions` 接口中有 `templateId` 和 `accountId` 字段，但没有 `workspaceId`。风格规则查询依赖于 `workspaceId`，如果 `prompt-assembler-service` 在组装提示词时不传递 `workspaceId`，可能导致规则匹配不到。

## 五、代码质量问题

### Q-1: 自动脚本注入导致代码结构异常

**问题描述**:  
使用 Python 脚本批量添加 `requireAuth` 导致部分文件结构异常：
- `requireAuth` 调用被插入到 `try {` 之后而不是之前
- 某些文件 `import { NextRequest }` 被遗漏（如 `suggest-opinion/route.ts` 之前只有 `Request`）
- 部分路由处理函数的缩进被打乱

**建议**: 手动审查所有脚本注入的文件，确保代码结构正确。

### Q-2: 错误处理不一致

**问题描述**:  
部分路由在 `requireAuth` 返回 401 时直接 `return`，但后续代码可能仍在执行（如 `try-catch` 结构不完整）。

### Q-3: `workspaceId` 兜底值 `'default-workspace'` 可能导致数据泄露

**问题描述**:  
`getWorkspaceId()` 在无法获取有效 workspace 时返回 `'default-workspace'`。如果某个用户的 session 过期但请求仍到达路由，所有用户可能共享同一个默认 workspace 的数据。

**建议**: 在生产环境中，如果无法确定 workspaceId，应返回 401 而非使用兜底值。

### 问题严重度分布

| 严重度 | 数量 | 关键问题 |
|--------|------|---------|
| P0 | 4 | 后端内部调用 401、数据流断裂、INSERT 字段错误、middleware 阻断 |
| P1 | 4 | WS 认证空表、事件不触发、参数语义混乱、重复函数 |
| P2 | 4 | 前端未用封装、签名混合、WS 加载无认证、prompt-assembler 未改造 |
| Q | 3 | 脚本注入异常、错误处理不一致、兜底值风险 |

### 修复优先级建议

1. **立即修复 P0-1 + P0-4**: requireAuth 增加 internal token 白名单，否则系统核心流程完全瘫痪
2. **立即修复 P0-2**: 所有 INSERT 操作补充 workspaceId，否则数据隔离形同虚设
3. **立即修复 P0-3**: style-deposition-service.ts 字段名修正，否则风格初始化不可用
4. **尽快修复 P1-1**: WebSocket 认证改为 JWT 验证
5. **尽快修复 P2-1**: 前端页面逐步切换到 apiGet/apiPost 封装
6. **后续清理**: P1-2/3/4、P2-2/3/4、Q-1/2/3

### 整体评价

本次改造的**架构设计方向正确**（Workspace 租户模型、RBAC 权限、认证中间件），但**实施层面存在系统性遗漏**：

1. **只做了"入口守卫"，没做"数据流通"**: 82 个路由加了 `requireAuth`，但 workspaceId 没有从入口传递到 service 层的 INSERT 操作。相当于门锁了但窗户开着。
2. **忽视了后端内部调用场景**: 系统大量使用后端 fetch 调用自身 API，这些调用不携带 session cookie，会被 401 阻断。
3. **前端集成不完整**: 创建了 `apiGet/apiPost` 封装但前端页面没有使用，WorkspaceSwitcher 事件机制不连贯。
4. **WebSocket 认证基于不存在的数据**: `accountSessions` 表在 JWT 策略下永远为空。

**建议**: 修复 P0 问题后，再进行一轮端到端测试，确保以下核心链路正常：
- 用户登录 → 任务拆解 → 子任务执行 → 文章生成 → 发布
- Agent 内部通信（B→D→C→B 闭环）
- 风格初始化 → 模板绑定 → 规则查询 → 提示词组装
