# 多平台发布功能 — 完整实施方案（v2.0 修正版）

> 基于技术评审报告修正，核心变更：从「交错 orderIndex」改为「独立 commandResultId」

## 一、需求定义

**核心目标**：用户输入一条指令，选择多个发布账号（跨平台），系统为每个平台生成不同风格的文章，发布时每个平台使用对应的文章版本。

**用户体验流程**：

```
用户输入指令
  → AI 拆解子任务
  → 创作引导区域：选择多个发布账号（如：公众号A + 小红书B）
  → 提交创建任务
  → 系统为每个账号创建独立的一组子任务（独立 commandResultId）
    ├─ group-wechat:  [insurance-d 写公众号文章] → [Agent B 评审] → [格式化] → [上传]
    └─ group-xhs:    [insurance-d 写小红书文章] → [Agent B 评审] → [格式化] → [上传]
  → 用户在发布面板查看各平台独立文章 → 确认发布
```

## 二、核心设计：独立 commandResultId

### 2.1 设计对比

| 维度 | 交错 orderIndex（v1 已否决） | 独立 commandResultId（v2 采纳） |
|------|------|------|
| 数据模型 | 共享 commandResultId + platformGroupIndex 区分 | 每个平台组独立 commandResultId |
| orderIndex 编排 | `groupIdx + stepIdx * total`（交错） | 每组独立从 1 开始（与现有逻辑完全一致） |
| 文章保存 | 会覆盖（P0 bug） | 天然隔离（不同 commandResultId） |
| split-publish | 需大改 | 零改动 |
| 大纲确认拆分 | 与多平台交互复杂 | 每组内部独立拆分，无冲突 |
| 执行引擎 | 需适配 platformGroupIndex | 零改动，且天然支持跨组并行 |
| 任务关联 | 通过 metadata.accountIds 松散关联 | 通过 metadata.multiPlatformGroupId 强关联 |

### 2.3 兼容性设计

| 场景 | 行为 |
|------|------|
| 只选1个账号 | 与改造前完全一致，单 commandResultId，无 multiPlatformGroupId |
| 选多个账号 | 每个账号独立 commandResultId，共享 multiPlatformGroupId |
| 旧数据（无 multiPlatformGroupId） | getTemplateIdForTask() 仍读取 metadata.accountId，不受影响 |
| 前端未升级的缓存 | selectedAccountId 字段保留兼容，优先使用 selectedAccountIds |

function buildPlatformContextPrefix(platformLabel: string, groupTotal: number): string {
  const platformGuidelines: Record<string, string> = {
    '微信公众号': '请按照微信公众号文章风格创作：标题≤64字，正文深度长文，HTML排版，专业语气',
    '小红书': '请按照小红书笔记风格创作：标题≤20字，正文≤1000字，语气亲切、多emoji、善用分段，重点突出干货',
    '知乎': '请按照知乎回答风格创作：标题为问题形式，正文专业深度，逻辑清晰，数据说话',
    '抖音': '请按照抖音视频脚本风格创作：标题≤30字，正文≤300字，口语化，节奏感强，开头抓人',
    '微博': '请按照微博正文风格创作：正文≤140字，话题标签，简洁有力，热点关联',
  };
  
  const guideline = platformGuidelines[platformLabel] || `请按照${platformLabel}的内容规范创作`;
  return `【目标平台：${platformLabel}】${groupTotal > 1 ? `（本文为${platformLabel}专属版本，与其他平台版本风格不同）` : ''}${guideline}\n\n`;
}

## 五、边界与约束

1. **不做并行执行优化**：多组子任务串行执行（引擎定时轮询时依次处理），降低引擎复杂度
2. **不做跨组子任务依赖**：每个平台组独立闭环，不依赖其他组的输出
3. **不做平台特化的 Agent**：仍使用 insurance-d，通过提示词注入平台风格
4. **平台数量上限**：前端限制最多选择 3 个平台
5. **单平台完全兼容**：选1个账号时与改造前行为完全一致

## 六、风险与缓解

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| 多组子任务串行执行耗时过长 | 低 | 引擎 processGroup 天然支持跨组并行，后续可启用 |
| 用户误选过多平台 | 低 | 前端限制最多3个，超出提示"建议分批创建" |
| LLM 某平台输出质量差 | 中 | 增加"单平台重新生成"能力（Phase 2） |
| 旧数据兼容性 | 低 | metadata.accountId 保留，新逻辑优先读 accountIds |
