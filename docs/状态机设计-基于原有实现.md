# 状态机设计 - 基于原有实现

## 1. 设计原则

**核心原则：基于原有实现，不添加原来功能没有的状态**

## 2. 原有状态定义

从原有实现中，我们使用以下状态：

| 状态 | 含义 | 说明 |
|------|------|------|
| `pending` | 待执行 | 任务已创建，等待执行 |
| `in_progress` | 执行中 | 任务正在执行 |
| `pre_completed` | 执行Agent完成，等待Agent B评审 | 执行Agent说搞定了，等待Agent B评审 |
| `pre_need_support` | 执行Agent需要帮助，等待Agent B评审 | 执行Agent需要帮助，等待Agent B评审 |
| `completed` | 已完成 | 任务执行成功 |
| `waiting_user` | 等待用户 | 需要用户介入确认/选择 |
| `failed` | 失败 | 任务执行失败 |
| `need_support` | 需要支持 | 需要人工支持 |
| `blocked` | 阻塞 | 任务执行阻塞 |
| `timeout` | 超时 | 任务执行超时 |
| `escalated` | 已升级 | 任务已升级，需要人工干预 |

## 3. 原有状态机流程

### 3.1 主流程

```
pending → in_progress → pre_completed 或 pre_need_support
pre_completed → Agent B评审 → completed 或 waiting_user 或 pending
pre_need_support → Agent B评审 → pending 或 waiting_user
```

### 3.2 详细流程

#### 3.2.1 pending → in_progress → pre_completed/pre_need_support

```
1. pending 状态
   ↓
2. 更新为 in_progress
   ↓
3. 调用执行Agent
   ↓
4. 根据执行Agent结果：
   - 如果执行Agent说搞定了 → pre_completed
   - 如果执行Agent需要帮助 → pre_need_support
```

#### 3.2.2 pre_completed → Agent B评审

```
1. pre_completed 状态
   ↓
2. 验证任务
   ↓
3. 记录 Agent B request
   ↓
4. 调用 Agent B
   ↓
5. 解析响应
   ↓
6. 处理评审结果
   ↓
7. 根据 Agent B 决策：
   - APPROVE → completed
   - NEED_REVISE → pending
   - NEED_USER → waiting_user
```

#### 3.2.3 pre_need_support → Agent B评审

```
1. pre_need_support 状态
   ↓
2. 验证任务
   ↓
3. 记录 Agent B request
   ↓
4. 调用 Agent B
   ↓
5. 解析响应
   ↓
6. 处理评审结果
   ↓
7. 根据 Agent B 决策：
   - CAN_HELP → pending
   - NEED_USER → waiting_user
```

## 4. 原有状态机特点

### 4.1 中间状态设计

使用 `pre_completed` 和 `pre_need_support` 作为中间状态：
- 执行Agent完成后，先进入 `pre_completed` 状态
- 执行Agent需要帮助后，先进入 `pre_need_support` 状态
- Agent B 评审后决定最终状态

### 4.2 Agent B 评审流程

1. 验证任务
2. 记录 Agent B request
3. 调用 Agent B
4. 解析响应
5. 处理评审结果
6. 最后更新状态

### 4.3 异常处理

- 异常处理时状态不回退，保持原状态
- 记录异常信息到 metadata
- 根据异常类型决定是否需要告警
