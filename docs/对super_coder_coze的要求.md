### 生成目标
基于TypeScript语言（适配Node.js环境），实现保险领域MCP调用场景下Agent B与insurance-d的交互逻辑，核心要求：
1. 最大化Agent B/insurance-d（LLM实例）的智能决策能力，贴合真实业务身份：
   - insurance-d：保险科普文章创作者（有创作决策/选择/建议权）；
   - Agent B：保险事业部技术总监（负责MCP落地、智能体领域能力提升）；
2. 所有业务决策（创作建议、MCP方案、能力提升方向）依赖LLM语义推理，弱化硬编码；
3. 领域知识（创作规则/案例、MCP最佳实践）通过“轻量化检索”供LLM使用；
4. 交互流程支持“领域能力进化”（创作案例沉淀、MCP优化建议落地）；
5. 仅保留必要硬编码（HTTP请求、数据库连接、类型定义），业务规则/决策全交给LLM。

### 核心角色定义（LLM实例，带业务身份+决策权）
1. insurance-d（保险科普文章创作者）：
   - 核心身份：保险科普内容创作负责人，拥有创作全流程决策/选择/建议权；
   - 核心智能能力：
     ① 语义理解：补全模糊的创作任务（如“写医疗险文章”→“写面向30-40岁人群的医疗险常见误区科普文，1500字，适配微信公众号发布”）；
     ② 创作决策：自主判断创作任务是否需要调用MCP能力（如是否需要调用“外网数据爬取MCP”获取最新医疗险政策、是否需要调用“微信发布MCP”）；
     ③ 场景识别：自主识别创作场景标签（如医疗险科普/重疾险科普、公众号发布/小红书发布）；
     ④ 创作建议：输出MCP调用的必要性建议（如“建议先调用外网数据爬取MCP获取2026年最新医疗险政策，提升文章时效性”）；
   - 输出类型：结构化JSON（字段严格定义）：
     {
       is_need_mcp: boolean;       // 是否需要调用MCP
       problem: string;            // 补全后的创作任务描述（含创作决策）
       domain_scene: string;       // 保险创作场景标签（如“医疗险科普-公众号发布”）
       capability_type: string;    // 建议调用的MCP能力类型（如“data_crawl/wechat_publish”）
       creation_suggestion: string;// 创作决策建议（体现决策权）
     }
   - 无硬编码规则：创作场景/ MCP必要性判断不写死枚举，靠LLM对保险科普创作领域的理解。

2. Agent B（保险事业部技术总监）：
   - 核心身份：保险事业部技术负责人，负责MCP能力落地、智能体领域能力提升；
   - 核心智能能力：
     ① 方案决策：基于insurance-d的创作需求，自主决策MCP调用最优方案（如选择哪个MCP接口、参数如何优化）；
     ② 参数生成：生成贴合保险领域的MCP调用参数（如公众号文章标题带“【保险科普】”前缀、爬取参数限定“2026年医疗险政策”）；
     ③ 避错优化：参考历史MCP调用失败案例，给出参数优化建议；
     ④ 能力提升：输出智能体（insurance-d）领域能力提升建议（如“建议insurance-d新增‘少儿医疗险’创作场景的知识储备”）；
   - 输出类型：结构化JSON（字段严格定义）：
     {
       api_address: string;                // MCP调用地址
       params: Record<string, any>;        // 结构化MCP参数（贴合保险领域）
       risk_tips: string;                  // MCP调用风险提示（如“爬取政策数据需注意版权，建议筛选官方来源”）
       capability_upgrade_suggestion: string; // 智能体领域能力提升建议
     }
   - 无硬编码参数模板：MCP参数生成不靠拼接字符串，靠LLM对保险MCP最佳实践的推理。

### 生成内容要求
1. 完整TS代码结构（含类型定义、模块化封装）：
   - 控制器调度模块（AgentScheduler.ts）：极简调度逻辑，仅负责LLM调用、知识库检索、MCP调用、能力提升建议沉淀，无业务规则硬编码；
   - LLM调用封装模块（LLMAgentClient.ts）：统一调用Agent B/insurance-d，支持动态传入“保险创作知识”“MCP最佳实践”等增量领域知识；
   - 知识库检索模块（DomainKnowledgeRetriever.ts）：轻量化检索（MySQL/JSON文件），仅返回当前创作场景相关的知识（如“医疗险科普创作规则”“微信发布MCP最佳实践”）；
   - 案例沉淀模块（CaseRepository.ts）：自动沉淀创作案例、MCP调用案例，同时沉淀Agent B的“能力提升建议”，供下次调用时增量投喂；
   - 类型定义文件（types.ts）：严格定义insurance-d/Agent B的输出类型，仅做类型校验（必要硬编码）。

2. 弱化硬编码的关键要求：
   - 不写死保险科普创作规则（如标题格式、内容禁忌）、MCP能力类型，而是通过“领域知识字符串”传给LLM，由LLM自主决策；
   - 不硬编码MCP参数模板，而是由Agent B（技术总监身份）基于创作需求自主设计参数；
   - 流程判断仅保留“MCP调用是否成功”的硬编码（如HTTP 200=成功），失败原因分析、能力提升方向交给LLM；
   - 智能体能力提升建议的落地逻辑仅做“建议存储”（硬编码），建议是否采纳、如何落地交给LLM。

3. 领域能力进化支持：
   - insurance-d创作任务执行成功后，自动沉淀“优质创作案例”（含创作决策、MCP调用方案）；
   - Agent B生成的“能力提升建议”自动存入知识库，下次调用insurance-d时增量投喂，推动其创作能力升级；
   - MCP调用失败时，Agent B自主分析失败原因并输出“MCP能力优化建议”，沉淀后推动MCP落地能力提升。

4. 技术适配：
   - 使用Node.js常用库（axios、mysql2、typescript），无小众依赖；
   - 所有代码添加完整TS类型注解，标注“业务决策逻辑（LLM负责）”vs“基础能力封装（TS负责）”；
   - 代码注释贴合业务身份（如“insurance-d创作决策”“Agent B技术总监方案”）。

### 输出格式
1. 完整可运行的TS代码（按模块拆分，含package.json依赖配置）；
2. 代码调用示例（demo.ts）：模拟“创作医疗险科普文”任务的完整执行流程；
3. 数据库/JSON文件结构：支持保险创作知识、MCP案例、能力提升建议的存储/检索；
4. 给insurance-d/Agent B的Prompt模板（贴合业务身份，体现决策权）；
5. 运行说明（安装依赖、启动方式、核心逻辑说明）。
