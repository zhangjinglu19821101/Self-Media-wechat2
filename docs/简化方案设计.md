# 简化方案设计：基于 Agent B 智能能力的简化

## 🎯 用户的核心观点（非常正确！）

1. ✅ **Agent B 是懂写代码的，很聪明**
2. ✅ **只要告诉它：是什么工具、什么方法、几个参数的含义**
3. ✅ **Agent B 完全能自己转换成调用参数**
4. ✅ **控制器只需要从 Agent B 返回中提取 tool_name、action_name、params，直接调用就行**
5. ✅ **不需要那么多复杂的字段**
6. ✅ **主要是细化 agent 的字段**

---

## 📋 简化后的 capability_list 字段设计

### 只保留关键字段！

| 字段 | 类型 | 必填 | 说明 | 示例 |
|------|------|------|------|------|
| `id` | serial | ✅ | 主键 | 11 |
| `capability_type` | text | ✅ | 能力类型 | "platform_publish" |
| `function_desc` | text | ✅ | 功能描述（给 Agent B 看） | "微信公众号-添加草稿" |
| `tool_name` | text | ✅ | 工具名 | "wechat" |
| `action_name` | text | ✅ | 方法名 | "addDraft" |
| `param_desc` | jsonb | ✅ | **参数含义说明**（新增！） | 见下方 |
| `status` | text | ✅ | 状态 | "available" |
| `metadata` | jsonb | ❌ | 元数据（可选） | `{}` |

## 🆕 核心新增字段：`param_desc`（参数含义说明）

### 设计思路
- **用自然语言描述每个参数的含义**
- **Agent B 一看就懂**
- **不需要复杂的 JSON Schema**
- **不需要模板**

## 📊 简化后的完整流程

### 1. 执行 Agent 输出能力边界判定

```json
{
  "is_need_mcp": true,
  "problem": "平台发布能力缺失，无微信公众号上传权限",
  "capability_type": "platform_publish"
}
```

---

### 2. 控制器查询 capability_list，给 Agent B 精简信息

**提示词示例**：
```
你需要调用 MCP 工具来完成任务。

可用的能力：

[能力 1]
ID: 11
功能描述: 微信公众号-添加草稿
工具名: wechat
方法名: addDraft
参数说明:
  - accountId: 微信公众号账号 ID，从 getAccounts 接口获取
  - articles: 文章数组，每个文章包含：
    - title: 文章标题，不超过 64 字
    - author: 作者，不超过 8 字（可选）
    - digest: 文章摘要，不超过 120 字（可选）
    - content: 文章正文，HTML 格式
    - thumb_media_id: 封面图片 media_id，从 uploadMedia 接口获取（可选）
    - ...其他参数...

[能力 2]
ID: 14
功能描述: 微信公众号-上传图片素材
工具名: wechat
方法名: uploadMedia
参数说明:
  - accountId: 微信公众号账号 ID，从 getAccounts 接口获取
  - mediaType: 素材类型，只能是 'image'（图片）或 'thumb'（缩略图）
  - fileUrl: 图片文件的 URL（fileUrl 和 fileBase64 二选一）
  - fileBase64: 图片文件的 Base64 编码（fileUrl 和 fileBase64 二选一）

请根据以上信息，选择合适的能力并生成调用参数。

返回格式要求：
{
  "solution_num": 11,
  "tool_name": "wechat",
  "action_name": "addDraft",
  "params": {
    "根据参数说明填充"
  }
}
```

### 4. 控制器直接调用（超级简单！）

```typescript
async function executeMCPCall(agentBOutput: any) {
  // 1. 直接提取三个字段
  const { tool_name, action_name, params } = agentBOutput;
  
  // 2. 直接调用！
  const tool = TOOL_MAP[tool_name];
  const method = tool[action_name];
  const result = await method(params);
  
  return result;
}
```

## 🎯 简化方案的优势

| 对比项 | 旧方案 | 新方案 |
|--------|--------|--------|
| **字段数量** | 10+ 个 | 6-7 个 |
| **param_desc** | ❌ 无 | ✅ 有（核心！） |
| **param_examples** | ✅ 有 | ❌ 不需要 |
| **param_template** | ✅ 有 | ❌ 不需要 |
| **interface_schema** | ✅ 有 | ❌ 不需要 |
| **agent_response_spec** | ✅ 有 | ❌ 不需要 |
| **Agent B 理解难度** | 中等 | ✅ 更容易（自然语言） |
| **维护成本** | 高 | ✅ 低（只维护 param_desc） |
| **扩展性** | 中等 | ✅ 更好（新增能力只需写 param_desc） |

---

## 📝 总结

### 核心要点

1. ✅ **相信 Agent B 的智能！** 它懂代码，给它自然语言描述就行
2. ✅ **只需要 3 个关键字段：** `tool_name`、`action_name`、`params`
3. ✅ **新增 `param_desc` 字段：** 用自然语言描述每个参数的含义
4. ✅ **删除复杂字段：** `param_examples`、`param_template`、`interface_schema`、`agent_response_spec` 都不需要
5. ✅ **控制器超级简单：** 直接提取字段调用就行
