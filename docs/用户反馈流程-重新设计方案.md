## 🎯 一、设计原则

### 1.1 核心原则

| 原则 | 说明 |
|------|------|
| **用户体验优先** | 用户交互简单、直观、流畅 |
| **状态机清晰** | 每个状态有明确的入口、出口、转换规则 |
| **可追溯性强** | 完整记录用户交互历史，支持审计 |
| **容错性强** | 支持重试、降级、边界情况处理 |
| **可扩展性好** | 预留扩展点，支持未来新增功能 |

## 🔍 二、当前流程问题分析

### 2.1 问题清单

| 问题 | 优先级 | 说明 |
|------|--------|------|
| **缺失历史记录恢复** | 🔴 高 | 用户反馈后，没有从 agent_sub_tasks_step_history 恢复历史状态 |
| **缺失 in_progress 状态处理** | 🔴 高 | 用户反馈后状态改为 in_progress，但执行引擎没有处理逻辑 |
| **缺失 RE_EXECUTE 决策类型** | 🔴 高 | Agent B 无法决策让执行 Agent 重新执行 |
| **状态机不完整** | 🟡 中 | 状态转换规则不够清晰 |
| **API 设计不够清晰** | 🟡 中 | 参数命名、返回结构可以优化 |

```
            │                │                │

    │  completed  │  │ waiting_user│  │   pending   │
    │  (已完成)   │  │  (继续等待) │  │  (重新执行)  │
    └─────────────┘  └─────────────┘  └──────┬──────┘
                                                   │
                                                   │ [控制器下次扫描]
                                                   │
                                                   ▼
                                          ┌─────────────┐
                                          │  in_progress │
                                          │  (执行中)    │
                                          └──────┬──────┘
                                                 │

                                                 │

                                    │  pre_completed 或    │
                                    │  pre_need_support    │
                                    └───────────────────────┘
```

---

### 3.2 状态详细定义

| 状态 | 触发场景 | 入口动作 | 出口动作 | 可转换到 |
|------|----------|----------|----------|----------|
| `waiting_user` | Agent B 决策需要用户介入 | 展示用户交互界面 | 用户提交反馈 | `user_feedback_received` |
| `user_feedback_received` | 用户提交反馈后 | 记录用户交互、解析反馈 | 执行引擎开始处理 | `in_progress` |
| `in_progress` (用户反馈后) | 执行引擎处理用户反馈 | 查询历史记录、恢复状态 | Agent B 决策完成 | `completed` / `waiting_user` / `pending` |
| `pending` (重新执行) | Agent B 决策 RE_EXECUTE | 清除 startedAt、保存重新执行指令 | 控制器下次扫描 | `in_progress` |

## 🔌 五、API 设计（重新设计）

### 5.1 提交用户反馈 API（v2）

**接口**: `POST /api/agents/user-feedback-v2`

**请求体**:

```typescript
{
  // 基础信息
  subTaskId: string;           // 必填：子任务 ID（agentSubTasks.id）

  feedbackType: 'select' | 'input' | 'confirm' | 'retry';  // 必填：反馈类型

  confirm?: boolean;           // feedbackType = 'confirm' 时必填

  userId?: string;             // 用户 ID
```

**响应体**:

```typescript
{
  success: boolean;

    subTaskId: string;
```

## 🎯 七、关键决策点（Agent B）

### 7.1 Agent B 决策类型扩展

| 决策类型 | 触发场景 | 后续状态 |
|---------|---------|---------|
| `COMPLETE` | 任务完成 | `completed` |
| `NEED_USER` | 继续需要用户 | `waiting_user` |
| `FAILED` | 任务失败 | `failed` 或 `waiting_user` |
| `EXECUTE_MCP` | 执行 MCP | 继续循环 |
| `RE_EXECUTE` | 让执行 Agent 重新执行 | `pending` ⭐ |

---

### 7.2 RE_EXECUTE 的 reasonCode

| reasonCode | 说明 |
|-----------|------|
| `USER_FEEDBACK_RE_EXECUTE` | 根据用户反馈要求重新执行 |
| `NEED_REVISE` | 执行结果需要修改 |
| `NEW_INFORMATION` | 有新信息需要重新执行 |
| `ADJUST_REQUIREMENTS` | 需求调整需要重新执行 |

## 🎉 十、总结

### 10.1 设计亮点

1. **清晰的状态机** - 每个状态有明确的职责
2. **完整的历史恢复** - 用户反馈后可以恢复完整的历史状态
3. **灵活的决策类型** - 新增 RE_EXECUTE 支持让执行 Agent 重新执行
4. **标准化的数据结构** - 交互记录格式统一，便于扩展
5. **渐进式实现** - 分三个阶段实现，降低风险
