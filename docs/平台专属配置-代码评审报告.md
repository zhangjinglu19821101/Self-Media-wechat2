**文件**: `src/app/api/agents/b/simple-split/route.ts`  
**位置**: 第 66 行

**问题描述**:
前端提交时传递了 `platformConfigs` 参数，后端解构接收但从未使用：

```typescript
// 前端传递
platformConfigs: Object.keys(platformConfigs).length > 0 ? platformConfigs : null,

// 后端接收
const {
  // ...
  platformConfigs, // 🔥🔥 按平台的专属配置

```

**影响**: 冗余代码，可能造成后续维护者困惑。

**修复建议**:
- 方案 A: 移除 `platformConfigs` 参数（因为配置已通过 `savePlatformConfig` 存储到数据库）
- 方案 B: 在子任务 metadata 中存储 `platformConfigs`，供后续使用

**推荐方案 A**，因为配置已在用户编辑时实时存储，无需重复传递。

#### #2 `savePlatformConfig` 缺少防抖

**文件**: `src/app/full-home/page.tsx`  
**位置**: 第 398-405 行

**问题描述**:
用户每次修改配置项都会立即触发 API 请求，无防抖处理：

```typescript
const savePlatformConfig = useCallback(async (accountId: string, platformConfig: any) => {
  try {
    await apiPut(`/api/platform-accounts/${accountId}`, { platformConfig });
  } catch (err: any) {
    console.error('保存平台配置失败:', err);
    toast.error('保存平台配置失败');
  }
}, []);
```

在 `select` 类型字段的 `onClick` 和 `text`/`textarea` 的 `onBlur` 中都直接调用：

```typescript
onClick={() => {
  // ... 更新状态
  savePlatformConfig(accountId, newConfig); // ❌ 无防抖
}}
```

**影响**: 
- `select` 类型：用户快速切换选项时产生多次请求
- 网络状况不佳时可能产生请求堆积

**修复建议**: 添加 500ms 防抖

```typescript
import { useRef, useCallback } from 'react';

// 在组件内部添加
const saveTimeoutRef = useRef<Record<string, NodeJS.Timeout>>({});

const savePlatformConfig = useCallback(async (accountId: string, platformConfig: any) => {
  // 清除之前的定时器
  if (saveTimeoutRef.current[accountId]) {
    clearTimeout(saveTimeoutRef.current[accountId]);
  }
  
  // 500ms 防抖
  saveTimeoutRef.current[accountId] = setTimeout(async () => {
    try {
      await apiPut(`/api/platform-accounts/${accountId}`, { platformConfig });
    } catch (err: any) {
      console.error('保存平台配置失败:', err);
      toast.error('保存平台配置失败');
    }
  }, 500);
}, []);
```

### P2 级别（建议优化）

#### #3 CollapsibleContent 默认折叠

**文件**: `src/app/full-home/page.tsx`  
**位置**: 第 2043-2060 行

**问题描述**:

#### #4 类型断言过于宽松

**文件**: `src/app/full-home/page.tsx`  
**位置**: 第 2039 行

**问题描述**:
```typescript
const currentConfig = (account.platformConfig as Record<string, any>)?.[account.platform] || {};
```

使用 `any` 类型断言，丢失类型检查。

**影响**: 类型安全问题，运行时可能出错。

**修复建议**: 使用正确的类型导入：

```typescript
import { PlatformConfig, PlatformType } from '@/lib/db/schema/style-template';

const currentConfig = (account.platformConfig as PlatformConfig)?.[account.platform as PlatformType] || {};
```

---

#### #5 错误处理不完善

**文件**: `src/app/full-home/page.tsx`  
**位置**: 前端配置保存逻辑

**问题描述**:
`savePlatformConfig` 失败后只显示 toast，但状态已经更新到 `accountConfigs`，造成 UI 和数据库不一致。

```typescript
// 先更新状态
setAccountConfigs(prev => prev.map(ac =>
  ac.account.id === accountId
    ? { ...ac, account: { ...ac.account, platformConfig: newConfig } }
    : ac
));
// 后保存（可能失败）
savePlatformConfig(accountId, newConfig); // 失败时状态已更新
```

**影响**: 用户看到已保存，实际数据库未更新。

**修复建议**: 
1. 保存成功后再更新状态，或
2. 失败后回滚状态
