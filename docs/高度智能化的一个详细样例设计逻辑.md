# 高度智能化的一个详细样例设计逻辑

## 一、业务场景说明

### 1.1 场景概述
本设计文档描述的是 **insurance-d → Agent B → 控制器 → 公众号上传 MCP 服务** 的标准化交互流程。

### 1.2 具体业务流程
1. **insurance-d 遇到能力阻塞**：insurance-d 在执行任务时，发现自己不具备公众号文章上传能力
2. **insurance-d 告知 Agent B**：insurance-d 将阻塞问题反馈给 Agent B
3. **控制器提供 MCP 清单**：控制器从 `capability_list` 表查询可用的 MCP 能力清单，提供给 Agent B
4. **Agent B 匹配能力**：Agent B 分析问题，匹配到可解决问题的公众号上传 MCP 能力
5. **Agent B 返回标准指令**：Agent B 不直接调用接口，只按照表中规范返回标准指令
6. **控制器识别并执行**：控制器识别指令 → 校验参数 → 调用后台 MCP 服务
7. **后台 MCP 完成上传**：后台 MCP 服务完成公众号文章上传，敏感信息（appid/secret）不对外暴露

---

## 二、数据库表设计

### 2.1 表名：capability_list

### 2.2 新增字段：agent_response_spec
| 字段名 | 字段类型 | 是否非空 | 默认值 | 字段用途 |
|--------|----------|----------|--------|----------|
| agent_response_spec | jsonb | 是 | {} | 存储每个 MCP 接口专属的 Agent B 返回格式规范 |

### 2.3 数据库执行 SQL
```sql
ALTER TABLE capability_list ADD COLUMN agent_response_spec jsonb NOT NULL DEFAULT '{}'::jsonb;
```

## 三、agent_response_spec 字段标准格式

### 3.1 完整格式示例
```json
{
  "trigger_key": "call_mcp_meth_status",
  "trigger_value": "true",
  "required_params": [
    {
      "param_name": "accountName",
      "param_type": "string",
      "example_value": "智者足迹 - 探寻",
      "desc": "公众号名称"
    },
    {
      "param_name": "content",
      "param_type": "string",
      "example_value": "我是一个英俊潇洒的伪君子",
      "desc": "公众号文章内容"
    }
  ],
  "response_example": {
    "call_mcp_meth_status": "true",
    "accountName": "智者足迹 - 探寻",
    "content": "我是一个英俊潇洒的伪君子"
  },
  "constraints": [
    "trigger_value 必须是字符串类型，不能用布尔 true/false",
    "必须返回所有 required_params 中的参数，不可缺失",
    "不允许携带任何敏感信息，如 appid、secret 等"
  ]
}
```

### 3.2 字段说明

#### 3.2.1 trigger_key
- **类型**：string
- **说明**：触发关键字，用于控制器识别是否需要调用 MCP
- **示例值**：`"call_mcp_meth_status"`

#### 3.2.2 trigger_value
- **类型**：string
- **说明**：触发值，必须是字符串类型，不能用布尔 true/false
- **示例值**：`"true"`

#### 3.2.3 required_params
- **类型**：array
- **说明**：必填参数列表，每个参数包含以下字段：
  - `param_name`：参数名
  - `param_type`：参数类型（string、number、boolean 等）
  - `example_value`：示例值
  - `desc`：参数描述

#### 3.2.4 response_example
- **类型**：object
- **说明**：Agent B 返回的完整示例

#### 3.2.5 constraints
- **类型**：array
- **说明**：约束条件列表

## 四、Agent B 必须返回的标准 JSON 格式

### 4.1 标准格式示例
```json
{
  "call_mcp_meth_status": "true",
  "accountName": "智者足迹 - 探寻",
  "content": "我是一个英俊潇洒的伪君子"
}
```

### 4.2 Agent B 返回规则
1. **必须包含 trigger_key**：`call_mcp_meth_status = "true"`
2. **必须返回所有 required_params 中的参数**：不可缺失
3. **不携带任何敏感信息**：如 appid、secret 等
4. **格式必须与 response_example 完全一致**

## 五、核心代码实现

### 5.1 Agent B 核心逻辑

#### 5.1.1 函数：generateAgentResponse
```typescript
/**
 * Agent B核心逻辑：读取agent_response_spec，生成标准化返回指令
 * @param mcpCapability 从capability_list表查询的单条MCP能力（仅用agent_response_spec字段）
 * @param businessData insurance-d传入的业务数据
 * @returns Agent B返回的标准化指令
 */
function generateAgentResponse(
  mcpCapability: { agent_response_spec: any },
  businessData: { accountName: string; content: string }
): any {
  const spec = mcpCapability.agent_response_spec;
  
  // 1. 初始化返回指令
  const response: Record<string, any> = {};
  
  // 2. 设置触发关键字和值（从spec读取）
  response[spec.trigger_key] = spec.trigger_value;
  
  // 3. 填充必填业务参数（从spec读取参数名）
  spec.required_params.forEach((param: any) => {
    const paramName = param.param_name;
    // 优先用业务数据，无则用示例值
    response[paramName] = businessData[paramName] || param.example_value;
  });
  
  // 4. 校验约束（保证指令合规）
  const isTriggerValid = typeof response[spec.trigger_key] === 'string';
  const isParamsComplete = spec.required_params.every((p: any) => !!response[p.param_name]);
  
  if (!isTriggerValid || !isParamsComplete) {
    throw new Error('返回指令不符合agent_response_spec约束');
  }
  
  return response;
}
```

#### 5.1.2 调用示例
```typescript
const mcpCapability = {
  agent_response_spec: JSON.parse(`{
    "trigger_key": "call_mcp_meth_status",
    "trigger_value": "true",
    "required_params": [
      {"param_name":"accountName","param_type":"string","example_value":"智者足迹-探寻"},
      {"param_name":"content","param_type":"string","example_value":"我是一个英俊潇洒的伪君子"}
    ]
  }`)
};

const businessData = { accountName: "智者足迹-探寻", content: "我是一个英俊潇洒的伪君子" };
const agentResponse = generateAgentResponse(mcpCapability, businessData);

console.log('Agent B返回指令：', agentResponse);
// 输出：{"call_mcp_meth_status":"true","accountName":"智者足迹-探寻","content":"我是一个英俊潇洒的伪君子"}
```

### 5.2 控制器核心逻辑

#### 5.2.1 函数：handleMcpCall
```typescript
import axios from 'axios';

/**
 * 控制器逻辑：校验指令 + 提取参数 + 调用后台MCP
 * @param agentResponse Agent B返回的指令
 * @param mcpCapability 从capability_list表查询的MCP能力（仅用agent_response_spec字段）
 */
async function handleMcpCall(
  agentResponse: any,
  mcpCapability: { agent_response_spec: any }
) {
  const spec = mcpCapability.agent_response_spec;
  
  // 步骤1：校验Agent B指令合规性（仅用spec字段）
  const validateResult = validateAgentResponse(agentResponse, spec);
  if (!validateResult.valid) {
    return { code: 400, msg: validateResult.msg };
  }

  // 步骤2：提取业务参数（从spec读取需要提取的参数名）
  const params: Record<string, string> = {};
  spec.required_params.forEach((param: any) => {
    params[param.param_name] = agentResponse[param.param_name];
  });

  // 步骤3：调用后台MCP接口（接口地址可硬编码/配置文件管理，无需表字段）
  const mcpUrl = '/api/upload-wechat-article'; // 后台固定接口地址
  try {
    const mcpResponse = await axios.post(mcpUrl, params);
    return { code: 200, msg: 'MCP调用成功', data: mcpResponse.data };
  } catch (err) {
    return { code: 500, msg: `MCP调用失败：${(err as Error).message}` };
  }
}
```

#### 5.2.2 函数：validateAgentResponse
```typescript
/**
 * 校验Agent B指令（仅依赖agent_response_spec）
 */
function validateAgentResponse(agentResponse: any, spec: any) {
  // 校验触发关键字和值
  if (!agentResponse[spec.trigger_key] || agentResponse[spec.trigger_key] !== spec.trigger_value) {
    return { valid: false, msg: `触发值错误，需为"${spec.trigger_value}"` };
  }
  
  // 校验必填参数
  for (const param of spec.required_params) {
    if (!agentResponse[param.param_name]) {
      return { valid: false, msg: `缺少必填参数：${param.param_name}` };
    }
    if (typeof agentResponse[param.param_name] !== param.param_type) {
      return { valid: false, msg: `${param.param_name}类型错误，需为${param.param_type}` };
    }
  }
  
  return { valid: true, msg: '指令合规' };
}
```

#### 5.2.3 调用示例
```typescript
const agentResponse = { call_mcp_meth_status: "true", accountName: "智者足迹-探寻", content: "我是一个英俊潇洒的伪君子" };
const mcpCapability = {
  agent_response_spec: JSON.parse(`{
    "trigger_key": "call_mcp_meth_status",
    "trigger_value": "true",
    "required_params": [
      {"param_name":"accountName","param_type":"string"},
      {"param_name":"content","param_type":"string"}
    ]
  }`)
};

handleMcpCall(agentResponse, mcpCapability).then(console.log);
```

## 六、控制器处理逻辑详细说明

### 6.1 处理流程
1. **检测触发关键字**：检测 Agent B 返回结果中是否存在 `call_mcp_meth_status = "true"`
2. **进入 MCP 调用流程**：存在则进入 MCP 调用流程
3. **读取参数规范**：从 `agent_response_spec` 中读取需要提取的参数
4. **提取参数**：从 Agent B 返回结果中提取参数
5. **调用后台接口**：调用后台 `/api/upload-wechat-article` 接口完成公众号文章上传

### 6.2 控制器处理流程图
```
insurance-d 遇到阻塞
    ↓
insurance-d 告知 Agent B
    ↓
控制器提供 MCP 清单（capability_list）
    ↓
Agent B 匹配能力，返回标准指令
    ↓
控制器检测：call_mcp_meth_status = "true"？
    ↓ 是
控制器校验参数
    ↓
控制器提取参数（accountName、content）
    ↓
控制器调用后台 MCP 接口
    ↓
后台 MCP 完成公众号文章上传
```

## 七、设计核心原则

### 7.1 Agent B 只负责返回标准指令
- Agent B 不直接调用接口
- Agent B 只按照 `agent_response_spec` 规范返回标准指令

### 7.2 敏感凭证全部封装在后端
- 敏感信息（appid、secret）不对外暴露
- 后台 MCP 服务负责处理敏感信息

### 7.3 所有返回规则配置化
- 返回规则存储在数据库字段 `agent_response_spec` 中
- 可扩展、可维护
- 一套规范适配所有 MCP 能力接口

### 7.4 弱化硬编码，最大化 LLM 智能决策
- 仅保留必要的硬编码（如参数校验）
- 业务规则和决策交给 LLM（Agent B）
- 接口地址可硬编码或配置文件管理，无需表字段

## 八、特殊场景说明

### 8.1 微信合规审核（content_audit）
微信合规审核是一个**特例**，不按照上述流程设计：
- Agent B 直接基于 RAG 检索 + LLM 给出审核结果
- 不需要控制器调用后台 MCP 接口
- 审核结果直接返回给执行 Agent（insurance-d）

## 九、总结

本设计文档详细描述了 **insurance-d → Agent B → 控制器 → 公众号上传 MCP 服务** 的标准化交互流程，核心要点包括：

1. **数据库表新增字段**：`capability_list.agent_response_spec`
2. **Agent B 返回标准指令**：基于 `agent_response_spec` 生成
3. **控制器校验并执行**：校验指令、提取参数、调用后台 MCP
4. **敏感信息后端封装**：不暴露给 Agent
5. **规则配置化**：可扩展、可维护

本设计遵循"弱化硬编码，最大化 LLM 智能决策"的原则，仅保留必要的硬编码（如参数校验），业务规则和决策交给 LLM（Agent B）。
