## 二、P0 级问题（必须解决）

### 🔴 P0-1：企业用户模式设计有根本性缺陷

**问题描述**：

方案中将企业定义为 `role='enterprise'` 的用户，同时通过 `user_members` 表关联成员。这种设计导致：

1. **身份二义性**：企业用户本身能否登录使用系统？还是仅作为容器？
2. **权限边界模糊**：企业 Owner 的个人数据 vs 企业数据如何区分？
3. **订阅计费复杂**：企业订阅费用算在 enterprise 用户上还是成员上？

**业务影响**：

- 企业用户购买 10 人套餐，Owner 自己创作算 1 个席位还是额外席位？
- 成员离职后，其创建的风格模板、素材是否随人走还是留企业？

**技术专家建议**：

```typescript
// 当前方案（有缺陷）
users.role = 'enterprise'  // 企业本身是一个用户

// 建议方案：组织（Workspace）与账户（Account）分离
// 表 1：accounts（账户表，每个自然人一个账户）
interface Account {
  id: string;
  email: string;
  name: string;
  type: 'personal' | 'member';  // 个人账户 或 企业成员账户
}

// 表 2：workspaces（组织表，企业/团队）
interface Workspace {
  id: string;
  name: string;           // 企业名称
  ownerId: string;        // 所有者账户ID
  plan: 'free' | 'pro' | 'enterprise';
  memberQuota: number;
  seatCount: number;      // 已购买的席位
}

// 表 3：workspace_members（成员关系）
interface WorkspaceMember {
  workspaceId: string;
  accountId: string;
  role: 'owner' | 'admin' | 'editor' | 'viewer';
  seatOccupied: boolean;  // 是否占用付费席位
}

// 所有资源归属 workspace，而非个人
// agent_tasks.workspace_id（非 user_id）
```

**业务专家建议**：

借鉴 SaaS 行业标准做法（Notion、Figma、Slack）：
1. 个人账户是核心，用户用个人账户登录
2. 创建/加入 Workspace，Workspace 是资源容器
3. 资源（任务、素材、模板）归属 Workspace，成员通过 Workspace 访问
4. 成员离职后，资源留在 Workspace，新成员可继承

**必须修订**：Phase 1 的用户模型设计需重新设计，改为 **Workspace 模式**。

### 🔴 P0-2：缺少配额与限流机制（业务核心）

**问题描述**：

方案中 `memberQuota` 仅作为字段存在，无实际配额控制逻辑。未考虑：

1. **创作额度限制**：个人免费版每月 10 篇，企业版每人每月 50 篇
2. **Agent 调用配额**：insurance-d 调用次数、token 消耗限制
3. **存储配额**：素材库容量限制（免费 100MB，付费 10GB）
4. **并发任务限制**：免费版同时只能运行 1 个任务

**业务影响**：

- 无法做 Freemium 商业模式
- 无法防止恶意刷量
- 企业版无法控制成本（成员无限制使用）

**建议方案**：

```typescript
// 新增 quota_usage 表，实时记录用量
interface QuotaUsage {
  workspaceId: string;
  period: string;          // 2026-02（按月）
  
  // 创作配额
  articlesCreated: number;
  articlesLimit: number;
  
  // Agent 调用配额
  agentCalls: Record<string, number>;  // { 'insurance-d': 45, 'agent-b': 12 }
  agentCallsLimit: number;
  
  // Token 消耗配额
  tokensConsumed: number;
  tokensLimit: number;
  
  // 存储配额
  storageUsed: number;     // bytes
  storageLimit: number;
  
  updatedAt: Date;
}

// 在每个关键操作前检查配额
async function checkQuota(workspaceId: string, operation: string) {
  const usage = await getQuotaUsage(workspaceId);
  if (usage.articlesCreated >= usage.articlesLimit) {
    throw new QuotaExceededError('本月创作额度已用完');
  }
}
```

**必须修订**：Phase 2 需增加配额体系设计，所有创作入口增加配额检查。

### 🔴 P0-4：WebSocket 认证方案不完整

**问题描述**：

方案中提到 "Phase 1.9 WebSocket 连接认证"，但未给出具体实现方案。

现有 WebSocket 实现（`src/lib/websocket-server.ts`）：
```typescript
// 当前无任何认证
wss.on('connection', (ws: WebSocket) => {
  ws.on('message', async (data) => {
    // 直接处理，不验证用户身份
  });
});
```

**安全风险**：

- 攻击者可连接 WebSocket 获取任意用户的任务状态推送
- 通过猜测 taskId 可订阅他人任务

**建议方案**：

```typescript
// 方案 A：Token 传递（推荐）
// 前端连接时携带 token
const ws = new WebSocket(`wss://api.example.com?token=${sessionToken}`);

// 服务端验证
wss.on('connection', (ws: WebSocket, req) => {
  const token = new URL(req.url, 'http://localhost').searchParams.get('token');
  const user = await verifyToken(token);
  if (!user) {
    ws.close(1008, 'Invalid token');
    return;
  }
  // 后续所有消息都关联 user.id
});

// 方案 B：Cookie 传递（如果 WS 与 HTTP 同域）
// 服务端从 req.headers.cookie 读取 session
```

**必须修订**：Phase 1.9 需给出完整 WebSocket 认证实现方案。

### 🟡 P1-2：迁移策略过于乐观

**当前方案问题**：

文档声称 "系统未上线，无需数据迁移"，但实际上：

1. **开发环境已有数据**：团队成员本地数据库已有测试数据
2. **表结构默认值陷阱**：
   ```sql
   ALTER TABLE agent_tasks ADD COLUMN user_id TEXT NOT NULL DEFAULT 'system-default-user';
   ```
   这会导致历史数据的 user_id 都是同一个值，失去隔离意义。

3. **外键约束冲突**：新增 `user_id` 字段后，原有数据若无对应用户会报错

**建议策略**：

```sql
-- 步骤 1：添加可为空的字段
ALTER TABLE agent_tasks ADD COLUMN user_id TEXT;

-- 步骤 2：清理或标记历史数据（开发环境）
-- 方案 A：删除所有历史数据（如果确实不重要）
DELETE FROM agent_tasks;

-- 方案 B：将历史数据分配给指定用户（管理员接管）
UPDATE agent_tasks SET user_id = 'admin-user-id' WHERE user_id IS NULL;

-- 步骤 3：改为 NOT NULL
ALTER TABLE agent_tasks ALTER COLUMN user_id SET NOT NULL;
```

### 🟡 P1-4：角色权限设计过于复杂

**当前设计**：

```typescript
enum Permission {
  TASK_CREATE, TASK_READ, TASK_UPDATE, TASK_DELETE,
  MATERIAL_CREATE, MATERIAL_READ, ... // 20+ 权限
}

const ROLE_PERMISSIONS = {
  [UserRole.ENTERPRISE_ADMIN]: [Permission.TASK_CREATE, ...],
  [UserRole.ENTERPRISE_EDITOR]: [Permission.TASK_CREATE, ...],
  // ... 维护成本高
};
```

**问题**：
- 20+ 权限点，5 种角色，权限矩阵维护困难
- 新增功能时需要更新权限矩阵，容易遗漏

**建议简化**：

```typescript
// 简化为 3 层权限模型
enum Role {
  OWNER = 'owner',    // 全部权限
  EDITOR = 'editor',  // 读写权限（不能删他人资源、不能管理成员）
  VIEWER = 'viewer',  // 只读权限
}

// 权限判断逻辑硬编码（简单清晰）
function canDelete(user: User, resource: Resource) {
  // Owner 可删全部
  if (user.role === Role.OWNER) return true;
  // Editor 只能删自己创建的
  if (user.role === Role.EDITOR) return resource.createdBy === user.id;
  // Viewer 不能删
  return false;
}
```

## 四、P2 级建议（可选优化）

### 🟢 P2-1：多平台发布优先级调整

当前方案将微信公众号作为 P0，但实际：

- 微信公众号申请门槛高（需要企业资质、300 元认证费）
- 小红书/知乎个人即可注册，更适合初期验证

**建议调整**：

| 平台 | 优先级 | 理由 |
|------|--------|------|
| 知乎 | P0 | 个人可注册，技术内容友好，API 相对开放 |
| 小红书 | P1 | 个人可注册，内容消费高 |
| 微信公众号 | P1 | 需企业资质，适合 B 端客户 |

```typescript
// Worker 独立进程或定时任务
setInterval(async () => {
  const pendingTasks = await db.select()
    .from(publishRecords)
    .where(
      and(
        eq(publishRecords.status, 'pending'),
        lte(publishRecords.scheduledAt, new Date())
      )
    )
    .limit(10);
  
  for (const task of pendingTasks) {
    await processPublish(task);
  }
}, 30 * 1000);
```

## 五、实施计划调整建议
