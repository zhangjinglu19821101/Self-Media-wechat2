# 9 步智能化流程分析

## 📋 完整流程回顾

### 一、agent_sub_tasks 模拟一条上传微信公众号文章的指令
### 二、执行agent（insurance-d） 任务超时
### 三、控制器带着标准询问模版，让执行agent（insurance-d）反馈遇到问题
### 四、执行agent按照标准化模版反馈
### 五、控制器将执行agent反馈的内容 + capability_list中的能力清单 + agent B的身份提示词提交给agent B
### 六、agent B判断系统具备解决问题的能力，通过输出标准化的调用MCP的上传微信公众号方法与相关参数组装
### 七、控制器解析agent B提供的工具、方法、参数，完成微信公众号的上传
### 八、控制器随后将第六步的内容 + 上传微信公众号方法上传结果提供给agent，让其将结果反馈给执行agent(insurance-d)
### 九、控制器依据insurance-d的返回，更新agent_sub_tasks对应的任务

## 🔍 问题分析

### 问题 1：缺少状态机定义 ⚠️

**问题描述**：
整个流程缺少明确的状态机定义，不知道每个步骤之间如何流转。

**需要定义的状态**：
```typescript
type SubTaskStatus = 
  | 'pending'           // 待处理
  | 'in_progress'       // 执行中
  | 'timeout'           // 超时（第二步）
  | 'asking_agent'      // 询问执行 agent（第三步）
  | 'agent_feedback'    // 执行 agent 反馈（第四步）
  | 'calling_agent_b'   // 调用 Agent B（第五步）
  | 'agent_b_response'  // Agent B 响应（第六步）
  | 'executing_mcp'     // 执行 MCP（第七步）
  | 'notifying_agent'   // 通知执行 agent（第八步）
  | 'completed'         // 完成（第九步）
  | 'failed';           // 失败
```

### 问题 9：缺少回滚和错误处理机制 ⚠️

**问题描述**：
如果某一步失败了，如何回滚？如何重试？

**需要考虑的场景**：
1. 询问执行 agent 失败（执行 agent 不响应）
2. Agent B 调用失败
3. MCP 执行失败
4. 通知执行 agent 失败

**建议的重试策略**：
```typescript
interface RetryStrategy {
  maxRetries: number;           // 最大重试次数，默认 3 次
  retryDelay: number;           // 重试延迟（毫秒），默认 1000
  backoffMultiplier: number;    // 退避倍数，默认 2
}
```
