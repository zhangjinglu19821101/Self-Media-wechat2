# 重要-开发流程注意事项

## 问题背景

在保险业务智能体系统中，公众号格式化任务（order_index = 4）出现死循环问题。

### 根本原因：
1. insurance-d 执行结果没有正确保存到 `agentSubTasks.result_text` 字段
2. `buildExecutionContext` 方法获取前序结果时，没有正确的文章内容
3. MCP 执行时缺少 title/content 参数，导致失败重试，陷入死循环

---

## 两个解决方案评估

### 方案1：所有 order_index 均更新 article_content 表

**优点：**
- article_content 表是专门为文章存储设计的
- 有完整的业务逻辑和字段
- 可以长期存储，不受任务状态影响

**缺点：**
- 影响范围大，需要修改多处逻辑
- article_content 表有自己的业务状态机，复杂度高
- 与 agentSubTasks 表已经有 result_text 字段，重复存储

**推荐度：** ❌ 不推荐

---

### 方案2：所有 order_index 均更新 result_text 字段

**优点：**
- ✅ **影响最小**：result_text 字段已经在 agentSubTasks 表中
- ✅ **逻辑已存在**：markTaskCompleted 等方法已经有保存逻辑
- ✅ **buildExecutionContext 已优化**：已修改为优先使用 result_text
- ✅ **单一职责**：每个 order_index 的执行结果直接保存在自己的记录中

**缺点：**
- 需要确保所有状态更新方法都处理 result_text

**推荐度：** ✅✅✅ **强烈推荐**

---

## 最终选择：方案2

### 决策理由
选择方案2（所有 order_index 均更新 result_text）

## buildExecutionContext 前序结果获取逻辑

### 从哪里获取？

**直接查询数据库：
1. 查询同一个 commandResultId 下的所有子任务
2. 找到 order_index = currentOrderIndex - 1 的前序任务
3. 优先使用该任务的 result_text 字段

### 存放于哪里？

`priorStepOutput` 变量 → 放入执行上下文 → 传递给后续执行

### 原字段的值是什么？

**是的！** `result_text` 字段应该是**每个指令最终执行结果的文本化版本：

| order_index | 执行者 | result_text 应该包含 |
|-------------|--------|---------------------|
| 1 | insurance-d | 完整的文章内容（纯文本） |
| 2 | T | 合规校验结果 |
| 3 | insurance-d | 修改后的文章内容 |
| 4 | T | 公众号格式化结果（HTML） |
| 5 | T | 上传结果 |

## 数据库表结构与 Drizzle Schema 不一致问题

### 问题现象

保存信息速记时报错：
```
null value in column "title" violates not-null constraint
null value in column "source_org" violates not-null constraint
null value in column "remind_status" violates not-null constraint
```

### 根本原因

**数据库表结构与 Drizzle Schema 定义不一致**

| 字段 | 数据库实际约束 | Schema 定义 | 问题 |
|------|---------------|-------------|------|
| `title` | **NOT NULL** | nullable | ❌ 约束冲突 |
| `source_org` | **NOT NULL** | nullable | ❌ 约束冲突 |
| `remind_status` | **NOT NULL** | nullable + default | ❌ 约束冲突 |
| `snippet_type` | **NOT NULL** | nullable + default | ❌ 约束冲突 |

### 根因分析

1. 数据库表是用旧版 Schema 创建的（title、source_org 有 NOT NULL 约束）
2. 后来 Schema 改成了 nullable，但数据库表结构没有同步
3. Drizzle INSERT 时不传这些字段，违反数据库 NOT NULL 约束

### 排查方法

**关键排查命令：对比数据库实际结构与 Schema 定义**

```bash
# 1. 查看数据库实际表结构
./db.sh table info_snippets

# 2. 查看 Drizzle Schema 定义
cat src/lib/db/schema/info-snippets.ts

# 3. 对比 Nullable 列是否一致
# 数据库: 查看 "Nullable" 列是否为 "not null"
# Schema: 查看字段是否有 .notNull() 修饰符
```

### 修复方案

**方案1：修改数据库表结构（推荐）**

如果 Schema 定义是正确的，用 SQL 移除数据库 NOT NULL 约束：

```sql
ALTER TABLE info_snippets ALTER COLUMN title DROP NOT NULL;
ALTER TABLE info_snippets ALTER COLUMN source_org DROP NOT NULL;
ALTER TABLE info_snippets ALTER COLUMN remind_status DROP NOT NULL;
ALTER TABLE info_snippets ALTER COLUMN snippet_type DROP NOT NULL;
```

**方案2：修改 Schema 定义**

如果数据库约束是正确的，修改 Schema 添加 `.notNull()` 修饰符：

```typescript
// 修改前
title: text('title'),
sourceOrg: text('source_org'),

// 修改后
title: text('title').notNull(),
sourceOrg: text('source_org').notNull(),
```

### 预防措施

1. **修改 Schema 后必须同步数据库**：使用迁移脚本或手动 ALTER TABLE
2. **开发环境可使用 Drizzle Push**：`drizzle-kit push` 自动同步
3. **生产环境必须使用迁移**：避免数据丢失
4. **定期对比 Schema 与数据库**：特别是修改了 Schema 定义后
