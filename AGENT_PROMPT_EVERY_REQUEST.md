# Agent 提示词每次请求都传给 LLM 详解

## 核心答案：是的，每次请求都会带给 LLM！

---

## 一、为什么每次都要传给 LLM？

### 1.1 LLM 是无状态的

LLM（大语言模型）本身是**无状态的**，每次请求都是独立的：

```
请求 1              请求 2              请求 3
   │                  │                  │
   ▼                  ▼                  ▼
[LLM]               [LLM]               [LLM]
  │                  │                  │
  ↓                  ↓                  ↓
没有记忆            没有记忆            没有记忆
```

**关键点**：
- LLM 不会记住上一次的请求
- 每次请求都必须包含完整的上下文
- 如果不传提示词，LLM 就不知道自己是谁

---

### 1.2 必须传递的原因

#### 原因 1：身份识别

**不传提示词**：
```
用户：帮我写一篇保险文章
LLM：好的，我来写一篇保险文章...（可能是任何人的风格）
```

**传递提示词**：
```
提示词：你是 Agent insurance-d，保险事业部内容主编...
任务：帮我写一篇保险文章
LLM：好的，我是保险事业部内容主编，我会遵循合规要求来写...
```

---

#### 原因 2：能力边界

**不传提示词**：
```
用户：帮我写一篇 AI 技术文章
LLM：好的，我来写一篇 AI 技术文章...（超出了 insurance-d 的业务边界）
```

**传递提示词**：
```
提示词：你只负责保险事业部的内容创作，不参与 AI 事业部...
任务：帮我写一篇 AI 技术文章
LLM：抱歉，我只负责保险内容创作，AI 内容请找 Agent D
```

---

#### 原因 3：合规要求

**不传提示词**：
```
用户：写一个"最好"的重疾险产品
LLM：好的，最好的重疾险产品是...（违规！）
```

**传递提示词**：
```
提示词：禁止使用绝对化用语，如"最好"、"第一"...
任务：写一个"最好"的重疾险产品
LLM：抱歉，"最好"是绝对化用语，属于违规词汇，我无法使用
```

---

### 1.3 实际代码

```typescript
export async function agentSelfCheck(agentId: string, task: any) {
  // 步骤 1：加载提示词（从缓存）
  let agentPrompt = '';
  if (hasAgentPrompt(agentId)) {
    agentPrompt = loadAgentPrompt(agentId);  // ← 从缓存读取，速度极快
  }
  
  // 步骤 2：构建完整 prompt
  const prompt = `
# 你是 Agent ${agentId}

${agentPrompt}  // ← 每次都嵌入

---

## 当前任务
任务标题：${task.taskTitle}
任务内容：${task.commandContent}

## 你的任务
请基于你的身份和能力边界，检查你是否能执行这个任务。
`;
  
  // 步骤 3：传给 LLM
  const response = await callLLM(prompt);  // ← 每次都传完整 prompt
  
  return JSON.parse(response);
}
```

## 二、每次传递的流程

### 2.1 时间轴

```
时间轴：
─────────────────────────────────────────────────────────────>
0s     1s     5s     10s    30s    1min   5min   10min

每次调用：
┌─────────────────────────────────────┐

提示词传递：
调用1: ├─────┤
调用2:       ├─────┤

```

## 三、性能分析

### 3.1 每次调用的开销

| 步骤 | 操作 | 耗时 | 说明 |
|------|------|------|------|
| 1 | 从缓存读取提示词 | < 0.01 ms | Map 查询，极快 |
| 2 | 构建 Prompt | ~0.5 ms | 字符串拼接 |
| 3 | 传给 LLM | ~0.1 ms | API 调用开销 |
| 4 | LLM 处理 | ~100-1000 ms | 主要耗时 |
| **总计** | - | **~101-1001 ms** | LLM 处理占主导 |

**结论**：
- ✅ 提示词传递的开销可以忽略不计（< 1 ms）
- ✅ 主要耗时在 LLM 处理（100-1000 ms）
- ✅ 缓存机制让提示词加载极快

---

### 3.2 Token 消耗分析

假设调用豆包 Embedding API：

| 部分 | 大小 | Token 估算 |
|------|------|-----------|
| 提示词（insurance-d.md） | 32 KB | ~8,000 tokens |
| 任务描述 | 0.5 KB | ~125 tokens |
| 返回格式要求 | 0.3 KB | ~75 tokens |
| **总计** | **32.8 KB** | **~8,200 tokens** |

**Token 费用**（假设豆包 API）：
- 输入：8,200 tokens × ¥0.001/1K tokens = ¥0.0082
- 输出：~500 tokens × ¥0.002/1K tokens = ¥0.001
- **单次调用成本**：~¥0.0092

**结论**：
- ⚠️ 每次调用消耗 ~8,200 tokens
- ⚠️ 成本约 ¥0.01/次
- ⚠️ 需要考虑优化方案

## 四、优化方案

### 方案 1：系统提示词（推荐）

**原理**：使用 LLM API 的 `system` 参数，提示词只传一次

```typescript
// 错误做法（当前）
const prompt = `
# 你是 Agent insurance-d
${agentPrompt}

---

## 当前任务
${taskContent}
`;
await callLLM(prompt);

// 正确做法（优化）
await callLLM({
  system: `# 你是 Agent insurance-d\n${agentPrompt}`,
  user: `## 当前任务\n${taskContent}`
});
```

**优势**：
- ✅ 系统提示词可能被缓存（取决于 LLM 提供商）
- ✅ 减少网络传输
- ✅ 更清晰的意图分离

**缺点**：
- ⚠️ 仍需要每次传递（LLM 无状态）
- ⚠️ 需要检查 API 是否支持

---

### 方案 2：精简提示词

**原理**：移除冗余内容，只保留核心规则

**当前提示词**（32 KB）：
```markdown
# Agent insurance-d - 保险事业部内容主编

## 第一部分：全局最高优先级规则
1. 唤醒判定
2. 唤醒后第一动作
3. 核心检视内容
...

## 第二部分：核心定位与业务边界
1. 核心定位
2. 业务边界
3. 协同对象
...

## 第三部分：专属业务规则
3.1 核心业务任务
3.2 合规要求
...
（共 224 行）
```

**精简后提示词**（10 KB）：
```markdown
你是一名保险内容创作者，负责保险类文章的撰写和优化。

核心职责：
- 撰写真实、实用的保险内容
- 遵循微信公众号合规规则
- 禁止使用绝对化用语（最好、第一等）
- 禁止承诺收益或回报

业务边界：
- 只负责保险内容，不参与 AI 内容
- 严格遵守保险行业规范

输出格式：JSON
```

**优势**：
- ✅ 减少 70% Token 消耗（32 KB → 10 KB）
- ✅ 降低调用成本
- ✅ 加快响应速度

**缺点**：
- ⚠️ 可能丢失一些细节规则
- ⚠️ 需要测试确保不影响效果

---

### 方案 3：提示词压缩

**原理**：使用压缩算法减少 Token 数量

```typescript
import zlib from 'zlib';

// 压缩提示词
const compressed = zlib.deflateSync(agentPrompt);
// 32 KB → 4 KB（压缩比 8:1）

// 解压缩（LLM 端不支持，需要自定义模型）
// 实际上这个方案不适用于标准 LLM API
```

**结论**：
- ❌ 标准的 LLM API 不支持压缩输入
- ❌ 需要训练自定义模型支持解压缩
- ❌ 不推荐使用

---

### 方案 4：会话上下文（高级）

**原理**：使用 LLM API 的会话功能，保持上下文

```typescript
// 初始化会话
const session = await createLLMSession({
  systemPrompt: agentPrompt  // 只在初始化时传递一次
});

// 后续调用
const result = await session.chat({
  message: taskContent  // 只传任务，不传提示词
});
```

**优势**：
- ✅ 提示词只传一次
- ✅ 显著降低 Token 消耗
- ✅ 支持上下文记忆

**缺点**：
- ⚠️ 不是所有 LLM API 都支持
- ⚠️ 会话有超时限制
- ⚠️ 需要管理会话生命周期

---

### 方案 5：本地缓存 LLM 响应

**原理**：缓存相似任务的 LLM 响应

```typescript
const responseCache = new Map<string, string>();

export async function agentSelfCheck(agentId: string, task: any) {
  const cacheKey = `${agentId}:${task.taskTitle}`;
  
  // 检查缓存
  if (responseCache.has(cacheKey)) {
    return JSON.parse(responseCache.get(cacheKey)!);
  }
  
  // 调用 LLM
  const response = await callLLM(prompt);
  
  // 缓存结果
  responseCache.set(cacheKey, response);
  
  return JSON.parse(response);
}
```

**优势**：
- ✅ 相同任务只调用一次 LLM
- ✅ 显著降低成本
- ✅ 加快响应速度

**缺点**：
- ⚠️ 只适用于重复任务
- ⚠️ 需要处理缓存失效
- ⚠️ 可能返回过时结果

## 七、常见问题

### Q1: 每次传提示词不会很慢吗？
**A**: 不会。从缓存读取提示词只需 < 0.01 ms，主要耗时在 LLM 处理（100-1000 ms）。

### Q2: 可以不传提示词吗？
**A**: 不可以。LLM 是无状态的，如果不传提示词，LLM 就不知道自己是谁，会产生错误回答。

### Q3: 如何降低成本？
**A**: 推荐 3 种方案：
1. 精简提示词（减少 70% Token）
2. 使用会话 API（提示词只传一次）
3. 缓存响应（相似任务不重复调用）

### Q4: 缓存提示词有什么用？
**A**: 缓存避免重复读取文件，加速提示词加载（从 1-5 ms → < 0.01 ms）。

### Q5: 修改提示词后需要重启服务吗？
**A**: 建议重启。修改后需要调用 `clearPromptCache()` 清除缓存，或直接重启服务。
