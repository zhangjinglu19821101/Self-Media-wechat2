# Agent 回执和状态反馈功能实现总结

## 功能概述

为所有执行端 Agent（B、C、D、insurance-c、insurance-d）实现标准化的回执和状态反馈功能，确保 Agent A 能够及时了解指令接收情况和任务执行状态。

## 核心功能

### 1. 指令接收即时回执

**要求**：
- 接收 Agent A 下达的任何指令/任务后，必须在1分钟内完成回执反馈
- 无客观障碍不得延迟
- 严格按固定格式执行

**固定格式**：
```
任务ID：【收到的任务唯一ID，无ID填「无」】
指令接收状态：【成功/失败（失败需标注原因）】
执行准备：【已明确核心要求，进入执行阶段/需补充核心信息】
```

### 2. 任务主动查询反馈

**要求**：
- 无条件支持 Agent A 发起的任何任务情况查询
- 收到 A 的查询指令后立即反馈（无延迟）
- 严格按固定格式执行

**固定格式**：
```
任务基础信息：任务ID【XXX】、任务名称【XXX】、接收时间【XXXX-XX-XX XX:XX】
当前执行状态：【未开始/执行中/已完成/暂停（暂停标注原因）】
核心完成进度：【量化进度】
已完成核心节点：【如「选题审核通过」】
待办核心事项：【如「剩余2篇文章18:00前完成」】
当前问题/异常：【无/客观障碍（标注具体障碍）】
```

## API 示例

### 生成任务接收回执（成功）

```bash
curl -X POST http://localhost:5000/api/agents/receipt \
  -H "Content-Type: application/json" \
  -d '{
    "type": "receipt",
    "params": {
      "taskId": "task-001",
      "status": "success"
    }
  }'
```

**响应内容**：
```
任务ID：【task-001】
指令接收状态：【成功】
执行准备：【已明确核心要求，进入执行阶段】
```

### 生成任务接收回执（失败）

```bash
curl -X POST http://localhost:5000/api/agents/receipt \
  -H "Content-Type: application/json" \
  -d '{
    "type": "receipt",
    "params": {
      "status": "failed",
      "failureReason": "指令无核心任务ID"
    }
  }'
```

**响应内容**：
```
任务ID：【无】
指令接收状态：【失败（失败原因：指令无核心任务ID）】
执行准备：【需补充核心信息】
```

### 生成任务状态反馈（执行中）

```bash
curl -X POST http://localhost:5000/api/agents/receipt \
  -H "Content-Type: application/json" \
  -d '{
    "type": "status-feedback",
    "params": {
      "taskId": "task-001",
      "taskName": "创作公众号文章",
      "receivedTime": "2026-02-03 14:30",
      "executionStatus": "in-progress",
      "progress": "内容创作完成8/10篇",
      "completedNodes": ["选题审核通过", "3篇文章上传草稿箱"],
      "pendingItems": ["剩余2篇文章18:00前完成"],
      "issues": "无"
    }
  }'
```

**响应内容**：
```
任务基础信息：任务ID【task-001】、任务名称【创作公众号文章】、接收时间【2026-02-03 14:30】
当前执行状态：【执行中】
核心完成进度：【内容创作完成8/10篇】
已完成核心节点：【选题审核通过、3篇文章上传草稿箱】
待办核心事项：【剩余2篇文章18:00前完成】
当前问题/异常：【无】
```

### 生成任务状态反馈（暂停）

当前执行状态：【暂停（公众号后台无法访问）】
核心完成进度：【问一问解答完成25条】
已完成核心节点：【无】
待办核心事项：【明日10:00前提交周复盘】
当前问题/异常：【客观障碍（公众号后台无法访问）】

## 执行状态说明

| 状态值 | 中文显示 | 使用场景 |
|--------|---------|---------|
| not-started | 未开始 | 任务已接收但尚未开始执行 |
| in-progress | 执行中 | 任务正在执行中 |
| completed | 已完成 | 任务已完成 |
| paused | 暂停 | 任务因客观障碍暂停（需标注原因） |

## 前端界面

### 回执和状态反馈管理面板

位置：执行端 Agent 页面底部

**标签页**：
1. **指令接收回执**
   - 任务 ID 输入框
   - 指令接收状态选择（成功/失败）
   - 失败原因输入框（仅失败时显示）
   - 生成回执按钮
   - 生成的回执显示区域（支持复制）

2. **任务状态反馈**
   - 任务 ID 输入框（必填）
   - 任务名称输入框（必填）
   - 接收时间输入框（必填，默认当前时间）
   - 当前执行状态选择（未开始/执行中/已完成/暂停）
   - 暂停原因输入框（仅暂停时显示）
   - 核心完成进度输入框（必填）
   - 已完成核心节点输入框（可选）
   - 待办核心事项输入框（可选）
   - 当前问题/异常输入框（必填）
   - 生成状态反馈按钮
   - 生成的状态反馈显示区域（支持复制）

## 使用流程

### 指令接收回执流程

1. Agent 接收到 Agent A 下达的指令/任务
2. 在1分钟内打开"回执和状态反馈管理"面板
3. 选择"指令接收回执"标签页
4. 填写表单信息
5. 点击"生成回执"按钮
6. 复制生成的回执，反馈给 Agent A

### 任务状态反馈流程

1. Agent 接收到 Agent A 的任务查询指令
2. 立即打开"回执和状态反馈管理"面板
3. 选择"任务状态反馈"标签页
4. 填写表单信息
5. 点击"生成状态反馈"按钮
6. 复制生成的状态反馈，反馈给 Agent A

## 优势

1. **标准化**：统一的回执和状态反馈格式，便于 Agent A 解析和处理
2. **及时性**：明确的时限要求（1分钟/立即），确保信息及时传递
3. **完整性**：涵盖所有必要信息，确保任务状态透明化
4. **灵活性**：支持多种执行状态和异常情况的处理
5. **易用性**：前端界面友好，支持一键复制
